What is a Loadable Kernel Module (LKM)?

🔌 The Core Idea

A Loadable Kernel Module is a piece of kernel code compiled as a separate binary (.ko file) that can be inserted into a running kernel without rebooting — and removed when no longer needed. This is how virtually every driver is shipped and deployed on Linux.

When you load a module with insmod, the kernel literally copies your code into kernel address space, resolves all symbol references (like printk), and calls your init function. From that moment, your code runs with full Ring 0 privilege — same as the core kernel itself.

✅ Why Modules (not built-in)?

  • Load on demand: Only load when the hardware is present
  • No reboot needed: Deploy, test, fix, reload instantly
  • Smaller kernel image: Fewer drivers compiled in = faster boot
  • Easier development: Change code → rmmod → insmod, done
  • Distribution friendly: One kernel, many driver packages

⚠️ The Price of Privilege

  • Your module runs in Ring 0 — full hardware access
  • A bug (NULL deref, use-after-free) causes a kernel oops or panic
  • No memory protection from the rest of the kernel — you share address space
  • No automatic resource cleanup if your init fails midway — you must do it
  • This is why we test in QEMU first

Your First Kernel Module — Complete Annotated Code

🎯
Read Every LineUnlike userspace Hello World, every single line of a kernel module has a reason. We'll go through each one in detail.
C — hello.c — Your First Kernel Module
/*
 * hello.c - The simplest possible kernel module
 * This file MUST be compiled against kernel headers, not glibc.
 */

/* ── REQUIRED HEADERS ─────────────────────────────────────────────────── */
#include <linux/init.h>     /* __init, __exit macros */
#include <linux/module.h>   /* MODULE_* macros, module_init/exit */
#include <linux/kernel.h>  /* pr_info(), KERN_INFO, printk() */

/* ── MODULE METADATA ──────────────────────────────────────────────────── */
/* These become readable via: modinfo hello.ko                             */
MODULE_LICENSE("GPL");              /* REQUIRED. "GPL" unlocks GPL-only symbols */
MODULE_AUTHOR("Your Name");         /* Shows in modinfo */
MODULE_DESCRIPTION("Hello World");  /* Brief description */
MODULE_VERSION("1.0");              /* Optional version string */

/* ── INIT FUNCTION ───────────────────────────────────────────────────── */
/* Called when the module is loaded (insmod / modprobe)
 * __init: tells linker to put this in .init.text section.
 *         After init succeeds, kernel FREES this memory.
 *         Never call an __init function after module loads!
 * Returns 0 on success, negative errno on failure.             */
static int __init hello_init(void)
{
    pr_info("Hello, Kernel World! Module loaded.\n");
    /* pr_info is shorthand for printk(KERN_INFO "...") */

    return 0;  /* 0 = success. Negative = failure (insmod returns error) */
}

/* ── EXIT FUNCTION ───────────────────────────────────────────────────── */
/* Called when module is unloaded (rmmod).
 * __exit: placed in .exit.text section.
 *         If module is built-in (not loadable), kernel discards this.
 *         Must undo EVERYTHING done in init, in reverse order.
 * No return value — cleanup MUST succeed.                       */
static void __exit hello_exit(void)
{
    pr_info("Goodbye, Kernel World! Module unloaded.\n");
    /* No return value. If cleanup is impossible, use BUG() or
     * mark module as permanent with try_module_get() */
}

/* ── REGISTER INIT/EXIT WITH THE KERNEL ──────────────────────────────── */
module_init(hello_init);  /* Macro: sets .init pointer in module struct */
module_exit(hello_exit);  /* Macro: sets .exit pointer in module struct */

🔍 Line-by-Line Deep Dive

#include <linux/module.h>
The single most important kernel header. Defines module_init(), module_exit(), MODULE_LICENSE(), EXPORT_SYMBOL(), THIS_MODULE, and the module structure itself. Without this, nothing compiles.
MODULE_LICENSE("GPL")
Mandatory. Sets the license. "GPL" is critical because many kernel functions are exported as EXPORT_SYMBOL_GPL() — only GPL modules can use them. Using "Proprietary" taints the kernel and blocks access to GPL-only symbols. Options: "GPL", "GPL v2", "GPL and additional rights", "MIT", "Proprietary".
static int __init
static: limits symbol visibility to this file (good practice). __init: a macro expanding to __attribute__((__section__(".init.text"))) — tells the linker to place this function in a special section that is freed after boot/module init succeeds. Saves memory.
pr_info()
Wrapper for printk(KERN_INFO "..."). Uses C99 variadic macros. Output goes to the kernel ring buffer (readable via dmesg). On development kernels, also appears on serial console.
return 0
Zero = success. Module is now "loaded". Any negative errno = failure — module is immediately unloaded, insmod returns an error to shell. The exit function is NOT called on init failure (you must clean up partially-initialized resources yourself before returning the error).
static void __exit
void: cleanup cannot fail. If it does, you have a memory leak or a dangling resource — but the kernel cannot refuse to unload. __exit: code placed in .exit.text section, discarded for built-in modules since they can never be unloaded.
module_init(hello_init)
Expands to: static initcall_t __initcall_hello_init = hello_init placed in .initcall section. The kernel's init code scans this section and calls all registered init functions. For modules, this pointer is stored in the struct module that insmod fills in.

The Makefile — Building Out-of-Tree Modules

📁 Project Structure

Create a directory for your module. You only need two files to start:

~/drivers/day2/ ├── hello.c ← your module source code ├── Makefile ← tells the kernel build system what to compile └── [after build:] ├── hello.ko ← the compiled kernel module (this is what insmod loads) ├── hello.o ← intermediate object file ├── hello.mod.c ← auto-generated: module metadata in C ├── hello.mod.o ← compiled metadata object ├── Module.symvers ← symbol version table (for EXPORT_SYMBOL) └── modules.order ← build order of modules
Makefile — The Standard Out-of-Tree Module Makefile
# Makefile for out-of-tree kernel module

# obj-m: tells kbuild to build this as a loadable module (.ko)
# obj-y would build it into the kernel image instead
obj-m += hello.o

# If your module has multiple source files, use this pattern:
# mydriver-objs := file1.o file2.o file3.o
# obj-m += mydriver.o

# KDIR: path to the kernel build tree for your TARGET kernel
# For the currently running kernel (most common during development):
KDIR ?= /lib/modules/$(shell uname -r)/build

# Default target: build the module
all:
	# $(MAKE): call make recursively
	# -C $(KDIR): change to the kernel build directory first
	# M=$(PWD): tell kbuild where our module source is (current dir)
	# modules: the target to build
	$(MAKE) -C $(KDIR) M=$(PWD) modules

# Clean target: removes all build artifacts
clean:
	$(MAKE) -C $(KDIR) M=$(PWD) clean

# Install target: copies .ko to /lib/modules and runs depmod
install:
	$(MAKE) -C $(KDIR) M=$(PWD) modules_install
	depmod -a

# NOTE: Indentation MUST be a TAB, not spaces!
⚠️
Makefile Indentation Must Be TABsMake requires TAB characters (not spaces) for command indentation. If you use spaces, you'll get: Makefile:10: *** missing separator. Stop. Check with cat -A Makefile — you should see ^I (TAB) before commands, not spaces.

🏗️ What Happens During Build — Under the Hood

When you run make, here's exactly what happens:

  1. Phase 1 — Main Makefile: Make enters your directory, reads obj-m += hello.o, then hands control to the kernel build system by running make -C /lib/modules/.../build M=$(PWD)
  2. Phase 2 — Kernel kbuild: The kernel's top-level Makefile takes over. It reads your Makefile again (now in the context of the kernel tree), finds hello.o, and compiles hello.c with the exact same compiler flags as the kernel was built with — including the correct -D__KERNEL__, include paths, etc.
  3. Phase 3 — Module metadata: kbuild auto-generates hello.mod.c containing your module's version magic string, parameter list, and symbol table. It compiles this too.
  4. Phase 4 — Linking: Links hello.o + hello.mod.o into hello.ko — an ELF relocatable object file (not a regular executable).

insmod vs rmmod vs modprobe — Know the Difference

Command What it Does Resolves Dependencies? When to Use
insmod hello.ko Directly inserts the specified .ko file into the kernel. Calls the kernel's finit_module() syscall. NO. If your module depends on another module (e.g., i2c-core), you must load that manually first or insmod fails with ENOENT (unknown symbol). During development when testing a single module. Fast, simple.
rmmod hello Removes the named module from the kernel. Calls delete_module() syscall. Fails if module is in use (use_count > 0). NO. Does not unload dependencies automatically. During development. Use module name, not filename.
modprobe hello Loads a module by name from /lib/modules/$(uname -r)/. Reads modules.dep and automatically loads all dependencies first. YES. Full dependency resolution. This is how the kernel auto-loads drivers when hardware is detected (udev triggers modprobe). Production use. After running make modules_install + depmod -a.
modprobe -r hello Removes a module and also unloads now-unused dependencies. YES. Full cleanup. Production use for clean removal.
bash — How modprobe Finds Modules
# After: sudo make modules_install && sudo depmod -a

# depmod scans /lib/modules/$(uname -r)/ and builds:
ls /lib/modules/$(uname -r)/
# modules.dep      ← dependency graph: which module needs which
# modules.alias    ← device ID aliases (PCI IDs, USB IDs → module name)
# modules.builtin  ← modules built into vmlinux (modprobe ignores these)

# Show what modprobe would load for a module:
modprobe --show-depends hello

# See module aliases (how kernel auto-selects driver for detected hardware)
modinfo e1000e | grep alias
# alias: pci:v00008086d000010BCs...
# The kernel sees PCI ID 0x8086:0x10BC → loads e1000e automatically

# Override module behavior via /etc/modprobe.d/
echo "options hello debug=1" | sudo tee /etc/modprobe.d/hello.conf

📋 What finit_module() Does Internally

When you run insmod hello.ko, the kernel's finit_module() syscall does all of this:

  1. Read the ELF: Parse hello.ko as an ELF relocatable object. Extract sections: .text, .data, .bss, .init.text, etc.
  2. Verify version magic: Check that hello.ko's vermagic string matches the running kernel exactly. If it doesn't match, insmod fails with EINVAL. This prevents loading modules built for a different kernel version.
  3. Allocate kernel memory: Allocate memory in kernel space (module_alloc) for the module's code and data.
  4. Copy and relocate: Copy sections into kernel memory and apply ELF relocations — patch all the symbol addresses (e.g., printk address).
  5. Resolve symbols: Look up all undefined symbols in the module against the kernel's symbol table (/proc/kallsyms). If any symbol is missing, fail with ENOENT.
  6. Run module notifiers: Notify other kernel subsystems (e.g., tracing) that a new module is loading.
  7. Call init function: Call your hello_init(). If it returns non-zero, module is immediately unloaded.
  8. Free __init sections: After successful init, free the .init.text memory (your __init function is gone forever).

📢 Log Levels — Full Reference

Every printk call has a log level that determines urgency. The kernel compares this against the current console log level to decide whether to print to the console immediately.

Level MacroNumericMeaningWrapper FunctionWhen to Use
KERN_EMERG"<0>"System is unusablepr_emerg()Imminent system crash. Very rare.
KERN_ALERT"<1>"Action must be taken immediatelypr_alert()Data corruption detected, hardware failure.
KERN_CRIT"<2>"Critical conditionspr_crit()Severe hardware or software failures.
KERN_ERR"<3>"Error conditionspr_err()Driver errors: failed init, hardware not responding.

C — All printk Forms + Best Practices
/* OLD style (still valid but verbose): */
printk(KERN_INFO "Driver loaded, version %d\n", version);
/* Note: KERN_INFO and the format string are adjacent — no comma!
   KERN_INFO expands to the string "<6>" which gets prepended */

/* MODERN style — prefer these wrappers: */
pr_err("Failed to allocate memory: %d\n", ret);
pr_warn("Deprecated API used by %s\n", current->comm);
pr_info("Device found: %s\n", dev_name);
pr_debug("Entering %s\n", __func__);  /* compiled out in production */

/* DEVICE-AWARE logging (use in drivers that have a struct device *dev):
   Prefixes output with device name — ALWAYS use these in real drivers! */
dev_err(dev, "Register read failed: %d\n", ret);
dev_warn(dev, "Link speed degraded\n");
dev_info(dev, "Probe successful\n");
dev_dbg(dev, "RX buffer: %d bytes\n", len);
/* Output: [  123.456] mydriver 0000:00:1f.0: Register read failed: -5 */

/* RATE-LIMITED logging (avoid log spam in hot paths): */
pr_err_ratelimited("DMA error occurred\n");  /* max 10 per 5 seconds */

/* ONCE logging (print only on first occurrence): */
pr_warn_once("This feature is deprecated\n");

/* Useful format specifiers unique to kernel: */
pr_info("Kernel pointer: %pK\n", ptr);   /* hides address unless CAP_SYSLOG */
pr_info("Function: %pS\n", ptr);        /* resolves to symbol name */
pr_info("IPv4: %pI4\n", &ipaddr);       /* prints 192.168.1.1 */
pr_info("UUID: %pUB\n", uuid);          /* prints formatted UUID */
pr_info("MAC: %pM\n", mac_addr);        /* prints aa:bb:cc:dd:ee:ff */
bash — Controlling the Log Level
# View current log levels
cat /proc/sys/kernel/printk
# Output: 4  4  1  7
#   [0] = current console log level (messages >= this level go to console)
#   [1] = default message log level (if no level specified in printk)
#   [2] = minimum console log level (can't set lower than this)
#   [3] = default console log level at boot

# Set console log level to DEBUG (7) — see ALL messages on console

# Enable debug for a specific file and line
sudo sh -c 'echo "file hello.c line 25 +p" > /sys/kernel/debug/dynamic_debug/control'

# Watch kernel messages in real-time
dmesg --follow --level=err,warn,info

Module Parameters — Runtime Configuration

⚙️ What are Module Parameters?

Module parameters let you configure your driver's behavior at load time without recompiling. They're passed via insmod on the command line, or set permanently in /etc/modprobe.d/. They're also exposed via sysfs at /sys/module/<name>/parameters/ and can sometimes be changed at runtime.

C — hello_params.c — Full Parameter Examples
#include <linux/init.h>
#include <linux/module.h>
#include <linux/kernel.h>
#include <linux/moduleparam.h>   /* module_param, MODULE_PARM_DESC */

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Your Name");
MODULE_DESCRIPTION("Module with parameters demo");

/* ── PARAMETER DECLARATIONS ──────────────────────────────────────── */

/* Integer parameter */
static int timeout = 100;                   /* default value: 100 */
module_param(timeout, int, 0644);          /* name, type, sysfs permissions */
MODULE_PARM_DESC(timeout, "Timeout in ms (default: 100)");
/*
 * module_param(name, type, perm)
 *   name  = variable name (must be already declared)
 *   type  = byte, short, ushort, int, uint, long, ulong, charp, bool, invbool
 *   perm  = sysfs file permissions:
 *           0644 = owner rw, group r, others r (most common for writable params)
 *           0444 = read-only from sysfs (cannot change after load)
 *           0    = not exposed in sysfs at all
 */

/* Boolean parameter */
static bool debug = false;
module_param(debug, bool, 0644);
MODULE_PARM_DESC(debug, "Enable debug logging (default: false)");

/* String parameter */
static char *device_name = "mydevice0";
module_param(device_name, charp, 0444);    /* read-only in sysfs */
MODULE_PARM_DESC(device_name, "Name of the device (default: mydevice0)");

/* Array parameter */
static int ports[4] = {80, 443, 8080, 8443};
static int ports_count = 4;               /* actual number of elements passed */
module_param_array(ports, int, &ports_count, 0644);
/* module_param_array(name, type, &nump, perm)
   nump = pointer to int that receives actual count of values passed */
MODULE_PARM_DESC(ports, "List of ports (comma-separated, max 4)");

static int __init hello_params_init(void)
{
    int i;

    pr_info("=== Module Parameters Demo ===\n");
    pr_info("timeout     = %d ms\n", timeout);
    pr_info("debug       = %s\n", debug ? "true" : "false");
    pr_info("device_name = %s\n", device_name);
    pr_info("ports count = %d\n", ports_count);

    for (i = 0; i < ports_count; i++)
        pr_info("  ports[%d] = %d\n", i, ports[i]);

    if (debug)
        pr_debug("Debug mode is active!\n");

    return 0;
}

static void __exit hello_params_exit(void)
{
    pr_info("Module with parameters unloaded\n");
}

module_init(hello_params_init);
module_exit(hello_params_exit);

EXPORT_SYMBOL — Sharing Functions Between Modules

🔗 Why Symbol Export Matters

By default, functions in a kernel module are private — other modules cannot call them. EXPORT_SYMBOL makes a function visible to other modules. This is how the kernel builds layered subsystems: i2c-core.ko exports i2c_transfer() so your sensor driver can call it.

There are two variants — choosing the wrong one is a common mistake with important consequences.

C — Library Module (exports symbols)
/* mylib.c — provides functions to other modules */
#include <linux/module.h>
#include <linux/kernel.h>
MODULE_LICENSE("GPL");

int mylib_add(int a, int b)
{
    pr_info("mylib_add(%d, %d)\n", a, b);
    return a + b;
}
/* Available to ALL modules (GPL or not) */
EXPORT_SYMBOL(mylib_add);

int mylib_secret(void)
{
    return 42;
}
/* Available ONLY to GPL-licensed modules */
EXPORT_SYMBOL_GPL(mylib_secret);
C — Consumer Module (uses exports)
/* myclient.c — uses mylib's exported symbols */
#include <linux/module.h>
#include <linux/kernel.h>
MODULE_LICENSE("GPL");

/* Declare extern — function is from another module */
extern int mylib_add(int a, int b);
extern int mylib_secret(void);

static int __init client_init(void)
{
    int result = mylib_add(3, 7);
    pr_info("3 + 7 = %d\n", result);
    pr_info("secret = %d\n", mylib_secret());
    return 0;
}
static void __exit client_exit(void) { }
module_init(client_init);
module_exit(client_exit);
MacroWho Can Use ItUse Case
EXPORT_SYMBOL(fn) Any module — GPL or proprietary Generic utility functions, hardware abstraction that should be broadly accessible. E.g., helper math functions.
EXPORT_SYMBOL_GPL(fn) Only GPL-licensed modules. Proprietary modules get ENOENT on load. Core kernel APIs, subsystem interfaces. Most new kernel exports use this. Forces proprietary driver authors to not depend on internal APIs.
EXPORT_SYMBOL_NS(fn, ns) Modules that declare MODULE_IMPORT_NS(ns) Namespaced exports — groups related symbols. E.g., EXPORT_SYMBOL_NS(fn, NVME). Newer API (kernel 5.4+)
⚠️
Load Order MattersIf myclient depends on mylib, you must insmod mylib.ko FIRST, then insmod myclient.ko. With modprobe, this is automatic via modules.dep. The dependency is tracked in Module.symvers — if you move module files around, keep this file with them.

vermagic — Version Magic Explained

🔏 Why Your .ko Won't Load on a Different Kernel

Every .ko file embeds a version magic string (vermagic) that must exactly match the running kernel. This is a safety mechanism — a module built for kernel 6.1 may use struct layouts or APIs that changed in kernel 6.8, causing silent memory corruption if loaded on the wrong version.

Terminal — Inspecting vermagic
$ modinfo hello.ko | grep vermagic
vermagic:       6.8.0-51-generic SMP preempt mod_unload modversions
#               │                │   │        │          └ CONFIG_MODVERSIONS=y (CRC checks)
#               │                │   │        └─ CONFIG_MODULE_UNLOAD=y
#               │                │   └────────── CONFIG_PREEMPT=y
#               │                └────────────── CONFIG_SMP=y (multi-core)
#               └─────────────────────────────── Exact kernel version string

# Try loading on wrong kernel:
$ sudo insmod hello.ko
insmod: ERROR: could not insert module hello.ko: Invalid module format

$ dmesg | tail -2
[  45.123] hello: version magic '6.8.0-51-generic SMP preempt mod_unload modversions' 
            should be '6.9.1-custom SMP preempt mod_unload modversions'

# Solution: ALWAYS build modules against the kernel they'll run on
# Point KDIR to the correct kernel headers
$ make KDIR=/lib/modules/6.9.1-custom/build

Module Reference Counting — Preventing Unsafe Removal

🔢 What is the Module Use Count?

Each module has a reference count (use count). rmmod fails with EBUSY if the use count is non-zero. This prevents removing a module that's actively being used — for example, you can't unload a filesystem module while a disk using that filesystem is mounted.

C — Managing Module Reference Count
#include <linux/module.h>

/* Increment ref count — prevents rmmod while this is held */
if (!try_module_get(THIS_MODULE)) {
    /* Module is being unloaded, reject new requests */
    return -ENODEV;
}

/* Do your work here... */

/* Decrement ref count — allow rmmod again */
module_put(THIS_MODULE);

/* Check current use count (for debugging): */
pr_info("Module use count: %d\n", module_refcount(THIS_MODULE));

/* In practice: char device drivers use this automatically via cdev.
   When userspace opens /dev/mydevice, kernel increments use count.
   When all fds are closed, it decrements. rmmod only succeeds when
   no files are open. We'll implement this in Day 8. */

C — practice_module.c — Full Project
/*
 * practice_module.c — Day 2 Practice Project
 * A kernel module with parameters that demonstrates the full module lifecycle
 */
#include <linux/init.h>
#include <linux/module.h>
#include <linux/kernel.h>
#include <linux/moduleparam.h>

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Your Name");
MODULE_DESCRIPTION("Day 2 Practice: Module with parameters");
MODULE_VERSION("1.0");

/* ── Parameters ──────────────────────────────────────────────── */
static char *greeting = "Hello";
module_param(greeting, charp, 0444);
MODULE_PARM_DESC(greeting, "Greeting string (default: Hello)");

static int count = 3;
module_param(count, int, 0644);
MODULE_PARM_DESC(count, "How many times to greet (default: 3)");

static bool verbose = false;
module_param(verbose, bool, 0644);
MODULE_PARM_DESC(verbose, "Enable verbose output (default: false)");

/* ── Init ─────────────────────────────────────────────────────── */
static int __init practice_init(void)
{
    int i;

    /* Validate parameters */
    if (count < 0 || count > 100) {
        pr_err("practice: count must be between 0 and 100, got %d\n", count);
        return -EINVAL;   /* Return error — init fails, module not loaded */
    }

    if (verbose)
        pr_info("practice: Loading module (verbose mode ON)\n");

    pr_info("practice: Parameters: greeting='%s', count=%d, verbose=%d\n",
            greeting, count, verbose);

    for (i = 1; i <= count; i++) {
        if (verbose)
            pr_info("practice: [%d/%d] %s, Kernel World!\n", i, count, greeting);
        else
            pr_info("practice: %s, Kernel World!\n", greeting);
    }

    pr_info("practice: Module loaded successfully\n");
    return 0;   /* success */
}

/* ── Exit ─────────────────────────────────────────────────────── */
static void __exit practice_exit(void)
{
    pr_info("practice: Module unloaded. Goodbye!\n");
}

module_init(practice_init);
module_exit(practice_exit);

Common Mistakes & How to Fix Them

ErrorMessage / SymptomCause & Fix
Wrong indentation Makefile:5: *** missing separator. Stop. Used spaces instead of TABs. Fix: sed -i 's/^ /\t/' Makefile
Missing kernel headers fatal error: linux/module.h: No such file or directory Install kernel headers: sudo apt install linux-headers-$(uname -r)
Version mismatch Invalid module format from insmod Module built for different kernel. Rebuild with KDIR=/lib/modules/$(uname -r)/build
Missing MODULE_LICENSE Module loads with taint warning. GPL-only symbols fail to resolve. Add MODULE_LICENSE("GPL");. Without it, kernel marks itself "tainted" and blocks GPL symbols.
Unknown symbol Unknown symbol foo (err -2) Module uses a function not exported by the kernel or dependency not loaded. Check with grep foo /proc/kallsyms.
Module in use rmmod: ERROR: Module practice_module is in use Something holds a reference. Check lsmod — the 3rd column is use count. Find who's using it with lsmod | grep practice (shows dependents).
Kernel taint [1234] hello: loading out-of-tree module taints kernel. Normal for development! Out-of-tree modules always taint. In production, sign your modules with CONFIG_MODULE_SIG.

What is the Linux Kernel? Core Concepts

🧠 The Kernel in One Sentence

The Linux kernel is the privileged software that runs closest to hardware. It mediates every interaction between user programs and physical resources — CPU time, RAM, storage, network, peripherals — and enforces security and isolation between them.

Think of it this way: your C program runs in a sandbox. When it needs anything real — a file, the screen, a socket — it asks the kernel. The kernel either does it (if allowed) or rejects the request.

🐧 Why Linux is Monolithic — with Loadable Modules

Linux is fundamentally monolithic: all kernel code shares a single address space and can call each other's functions directly (no IPC overhead). This is why Linux is blazing fast.

But Linux adds a key feature: Loadable Kernel Modules (LKMs). You can write a driver as a separate .ko file and load it at runtime with insmod. This gives the flexibility of a microkernel without the performance penalty. As a driver developer, you will write LKMs every single day.

Kernel Space vs User Space — The Most Critical Concept

⚠️

🔵 User Space

  • Privilege: Ring 3 (x86 CPL=3). Lowest privilege level.
  • Memory: Only sees its own virtual address space. Cannot see other processes or kernel memory.
  • Hardware: Cannot directly access ports, IRQs, DMA, MMIO registers.
  • Crash impact: Only that process dies. OS survives.
  • Examples: bash, nginx, your C application

🔴 Kernel Space

  • Privilege: Ring 0 (x86 CPL=0). Highest privilege level.
  • Memory: Can map and access any physical memory address.
  • Hardware: Can execute privileged instructions: cli, sti, in, out, load CR3, etc.
  • Crash impact: Kernel oops or panic. Whole system may freeze.
  • Examples: Scheduler, drivers, network stack, VFS

🔒 What Actually Enforces the Separation?

The separation is NOT just software — it is enforced by CPU hardware through two mechanisms working together:

  • Privilege Rings (x86) / Exception Levels (ARM): The CPU tracks the current privilege level in a hardware register (CS register on x86, bits[1:0] are CPL). When code runs at CPL=3, executing a privileged instruction like cli causes a General Protection Fault (#GP). The OS handles this by killing the process.
  • Page Tables + MMU: Every page in the page table has a User/Supervisor bit. Pages mapped with Supervisor bit set (kernel pages) are inaccessible to Ring 3 code. Even if user code tries to read a kernel address, the MMU raises a Page Fault (#PF) — the kernel handles this with a SIGSEGV to the process.
  • SMEP (Supervisor Mode Execution Prevention): Prevents the CPU from executing userspace pages while in Ring 0. Stops kernel exploits that redirect execution to shellcode in userspace.
  • SMAP (Supervisor Mode Access Prevention): Prevents Ring 0 from reading/writing userspace pages accidentally. This is why your driver MUST use copy_to_user()/copy_from_user() — these temporarily disable SMAP via stac/clac instructions while doing the copy safely.

The Syscall Mechanism — How User Space Talks to Kernel

📞 What is a System Call?

A syscall is the controlled, hardware-enforced gateway from Ring 3 to Ring 0. It's the ONLY legitimate way for userspace to request kernel services. There are ~400 syscalls in Linux (read, write, open, socket, mmap, etc.).

User Process (Ring 3) Kernel (Ring 0) ───────────────────────────────────────────────────────────────────── int main() { ┌─────────────────────────────┐ write(1, "hi\n", 3); │ sys_write() handler │ │ │ │ │ 1. Put args in registers │ 4. Validate args │ │ rax = 1 (syscall #1) │ 5. Write to fd 1 (stdout) │ │ rdi = 1 (fd) │ 6. Return result in rax │ │ rsi = "hi\n" (buf) │ │ │ rdx = 3 (len) └─────────────────────────────┘ │ ▲ │ 2. Execute SYSCALL instruction │ ├─────────────────────────────────────────────┤ │ CPU switches to Ring 0 │ │ Saves user RIP, RSP to TSS │ │ Loads kernel RSP │ │ Jumps to kernel entry point │ │ (entry_SYSCALL_64 in entry_64.S) │ │ │ │ 3. Kernel looks up syscall table │ │ sys_call_table[rax] → sys_write │ └─────────────────────────────────────────────┘ │ 7. SYSRET instruction │ CPU switches back to Ring 3 │ Restores user RIP, RSP │ Returns result from rax to user code ▼ result = 3 (bytes written) }
💡
Trace Any Program's SyscallsRun strace ./your_program to see every syscall it makes. This is essential for debugging driver interactions from userspace. Try strace ls right now and observe all the openat, read, close calls.
C — How the C Library Wraps Syscalls (conceptual)
// What glibc's write() actually does internally:
ssize_t write(int fd, const void *buf, size_t count) {
    long result;
    // Inline assembly: set up registers and invoke SYSCALL instruction
    asm volatile(
        "syscall"
        : "=a"(result)
        : "0"(__NR_write), /* rax = 1 (syscall number) */
          "D"(fd),       /* rdi = fd */
          "S"(buf),      /* rsi = buf pointer */
          "d"(count)     /* rdx = count */
        : "rcx", "r11", "memory"
    );
    return result;
}

// __NR_write is defined in <asm/unistd.h>:
// #define __NR_write 1

// Kernel receives it at entry_SYSCALL_64 (arch/x86/entry/entry_64.S)
// Then dispatches to sys_call_table[1] = sys_write()
// Defined in: fs/read_write.c → SYSCALL_DEFINE3(write, ...)

⚡ Why Syscalls Are "Expensive"

Every syscall involves: saving registers, switching privilege level, potentially flushing TLB (if KPTI is enabled — Meltdown mitigation), running kernel code, then reversing all that. A single syscall can cost 50–1000 nanoseconds depending on the system and what it does.

As a driver developer, you design interfaces to minimize unnecessary syscalls — for example, using mmap to give userspace direct access to a device's memory buffer rather than doing thousands of small read() calls.

Kernel Build System — Kconfig, Makefiles, .config

🔧 The Three Key Build System Files

  • Kconfig: Defines available configuration options and their dependencies. Written in a special Kconfig language. This is what make menuconfig reads to show you options.
  • .config: The actual configuration for your build — generated by running menuconfig or other config tools. Contains thousands of CONFIG_FOO=y/m/n entries.
  • Makefile: Every directory has one. Tells the build system which .c files to compile and under what conditions. Uses the obj-y (built-in) and obj-m (module) variables.
Kconfig — Driver Option Example
# drivers/char/Kconfig

config MY_SENSOR_DRIVER
    tristate "My Sensor Driver"
    depends on I2C
    select REGMAP_I2C
    help
      Say Y to build this driver into the kernel,
      M to build as a loadable module,
      N to skip it entirely.

      If unsure, say M.
Makefile — Driver Build Rule
# drivers/char/Makefile

# obj-y  = always built into kernel
# obj-m  = built as loadable module
# obj-n  = not built (or just omit)

obj-$(CONFIG_MY_SENSOR_DRIVER) += my_sensor.o

# If driver has multiple source files:
my_sensor-objs := my_sensor_core.o \
                  my_sensor_i2c.o \
                  my_sensor_spi.o
.config — Generated Configuration (excerpt)
#
# Linux/x86 6.x.y Kernel Configuration
#
CONFIG_X86=y
CONFIG_64BIT=y
CONFIG_SMP=y                 # Symmetric Multi-Processing
CONFIG_MODULES=y             # Enable loadable module support
CONFIG_MODULE_UNLOAD=y       # Allow unloading modules
CONFIG_I2C=m                 # I2C as a module
CONFIG_MY_SENSOR_DRIVER=m    # Our driver as a module
# CONFIG_STAGING is not set   # Disabled options look like this

Build the Kernel

bash
# -j$(nproc): use all CPU cores. CRITICAL for speed.
make -j$(nproc)

# This produces:
#   arch/x86/boot/bzImage  — compressed kernel image
#   vmlinux                — uncompressed kernel (for GDB debugging)
#   *.ko files             — kernel modules

# Build time:
#   Full build with all modules: 30-90 minutes
#   Minimal QEMU config: 5-15 minutes
#   After first build, incremental: seconds

# Build only modules (once kernel is built):
make modules -j$(nproc)

# Install modules to /lib/modules/<version>/ (your running system)
sudo make modules_install
💡
Speed tip: Add CONFIG_LOCALVERSION="-mykernel" to .config so your built kernel doesn't conflict with your distro's installed kernel. Also set CONFIG_DEBUG_INFO_NONE=y if you don't need debugging symbols — cuts build time and size dramatically.

Create a Minimal Root Filesystem for QEMU

mount -t proc none /proc

exec /bin/sh

find . | cpio -H newc -o | gzip > ../rootfs.cpio.gz

Boot Your Kernel in QEMU

bash
# Basic QEMU boot — you'll see kernel startup messages then a shell
qemu-system-x86_64 \
    -kernel linux-6.9/arch/x86/boot/bzImage \
    -initrd busybox-1.36.1/rootfs.cpio.gz \
    -nographic \
    -append "console=ttyS0 init=/init" \
    -m 512M

qemu-system-x86_64 \
    -enable-kvm \
    -kernel linux-6.9/arch/x86/boot/bzImage \
    -initrd rootfs.cpio.gz \
    -nographic \
    -append "console=ttyS0 init=/init nokaslr" \
    -m 1G \
    -smp 2 \
    -cpu host

# To exit QEMU: Ctrl+A then X

# For driver development — share a directory from host to guest:
qemu-system-x86_64 \
    -enable-kvm \
    -kernel bzImage \
    -initrd rootfs.cpio.gz \
    -nographic \
    -append "console=ttyS0 init=/init" \
    -m 1G \
    -virtfs local,path=/home/user/drivers,mount_tag=drivers,security_model=none
# In guest: mount -t 9p drivers /mnt/drivers

Kernel Startup — What Happens at Boot

Power On / UEFI/BIOS │ ▼ Bootloader (GRUB/U-Boot) • Loads bzImage into RAM • Sets up boot parameters (command line, initrd address) • Jumps to kernel entry point │ ▼ arch/x86/boot/header.S → startup_32 / startup_64 • Decompress the kernel (it's a self-decompressing archive) • Set up initial page tables (identity mapping) • Switch to 64-bit mode (if not already) │ ▼ arch/x86/kernel/head_64.S → start_kernel() [init/main.c] • This is the first C function that runs • Calls in order: setup_arch() ← detect CPU, memory map, DMI build_all_zonelists() ← set up NUMA memory zones page_alloc_init() ← initialize page allocator trap_init() ← set up IDT (interrupt descriptor table) mm_init() ← initialize memory management sched_init() ← initialize scheduler workqueue_init_early() ← set up workqueues init_IRQ() ← initialize interrupt subsystem softirq_init() ← set up softirq vectors timekeeping_init() ← initialize time late_time_init() ← start hrtimers calibrate_delay() ← measure loops_per_jiffy console_init() ← initialize serial console kmem_cache_init() ← initialize SLUB allocator ... rest_init() ← spawn PID 1 (init) and kthreadd │ ▼ PID 0: idle task (swapper) — runs when nothing else is ready PID 1: kernel_init() → execve("/sbin/init") — becomes your init/systemd PID 2: kthreadd — parent of all kernel threads (kworker, etc.)
💡
Read start_kernel() in init/main.c — it's the most educational file in the kernel tree. Every line is a fundamental subsystem being initialized. Spend 20 minutes reading it and you'll understand the kernel's structure better than 90% of people.

Key Kernel Concepts Every Driver Dev Must Know

📋

Process Context

Code running on behalf of a user process (syscall, driver read/write). Can sleep, can access current, can call mutex_lock().

⚡

Interrupt Context

Code running in an IRQ handler. Cannot sleep, cannot call scheduler, cannot acquire mutex. Must be FAST. Use spinlocks only.

🔄

Preemption

Kernel can be preempted (CONFIG_PREEMPT). Your driver code can be interrupted at any point. Protect shared state with locks.

🧠

current Pointer

current is a per-CPU pointer to the currently running task_struct. Valid in process context. Meaningless in interrupt context.

🗺️

Physical vs Virtual

Drivers work with virtual addresses. Hardware DMAs to physical addresses. Use virt_to_phys() / phys_to_virt() to convert.

🔗

Reference Counting

Kernel objects use kref / atomic_t refcounts. Object is freed when refcount hits 0. Prevents use-after-free bugs.

Exploring Key /proc and /sys Files Today

🔍 Commands to Run Right Now and Understand

Before writing a single line of driver code, get comfortable navigating the live kernel's data. Open a terminal and run each of these:

bash — Kernel Exploration Commands
# ── Kernel version and build info ──────────────────────────────────
uname -a                    # Full kernel version string
cat /proc/version           # Kernel version + compiler used
cat /proc/cmdline           # Boot parameters passed to kernel

# ── CPU and architecture info ───────────────────────────────────────
cat /proc/cpuinfo           # CPU details: flags show supported features
grep flags /proc/cpuinfo | head -1 | tr ' ' '\n' | grep -E "smep|smap|kvm"
# Look for: smep, smap, vmx (Intel VT-x), svm (AMD-V)

# ── Memory layout ──────────────────────────────────────────────────
cat /proc/meminfo           # RAM usage: total, free, cached, buffers
cat /proc/iomem             # Physical memory map: RAM, MMIO regions
cat /proc/ioports           # x86 I/O port assignments
cat /proc/vmstat            # VM statistics: page faults, swaps, etc.

# ── Devices and drivers ────────────────────────────────────────────
cat /proc/devices           # Registered char and block devices (major numbers)
cat /proc/modules           # Loaded modules (same as lsmod)
ls /sys/bus/                # All registered buses: pci, usb, i2c, spi, ...
ls /sys/class/              # Device classes: net, block, tty, input, ...

# ── Interrupt information ───────────────────────────────────────────
cat /proc/interrupts        # IRQ assignments and counts per CPU
cat /proc/softirqs          # Softirq statistics

# ── Live kernel messages ────────────────────────────────────────────
dmesg | tail -50            # Last 50 kernel messages
dmesg -T | grep -i error    # Kernel errors with timestamps
dmesg --follow              # Live stream of kernel messages (Ctrl+C to stop)

# ── Syscall tracing ─────────────────────────────────────────────────
strace -c ls /tmp           # Show syscall count/time summary for ls
strace -e trace=open,read,write cat /etc/hostname  # Trace specific syscalls

# ── Kernel symbol table ─────────────────────────────────────────────
sudo cat /proc/kallsyms | head -20   # All kernel symbols with addresses
sudo cat /proc/kallsyms | grep " T " | grep printk  # Find printk address
💡
Important: /proc/kallsymsThis file maps every kernel function and variable to its virtual address. As a driver developer, when you get an oops with a memory address, you cross-reference it against kallsyms to find which function crashed. This is step 1 of kernel debugging.

Kernel vs Userspace Programming Differences — Mental Shift

Topic Userspace C Kernel C
Standard Library stdio.h, stdlib.h, string.h, glibc NO libc. Use linux/kernel.h, linux/slab.h. Kernel has its own implementations.
printf / logging printf(), fprintf() printk(KERN_INFO "...") or pr_info("...")
Memory allocation malloc(), calloc(), free() kmalloc(size, GFP_KERNEL), kfree(). No virtual memory growth on fault.
Stack size Typically 8 MB, grows on demand FIXED 8 KB (or 16KB). Stack overflow = kernel oops. Never allocate large arrays on stack.
Floating point Fully supported NOT allowed in interrupt context. Only in process context after special setup. Avoid entirely.
Error handling Return -1 and set errno Return negative errno value: -EINVAL, -ENOMEM, -EIO. Use IS_ERR() / PTR_ERR().
Crash impact Only your process dies Kernel oops or panic — system may reboot. One NULL dereference can kill everything.
Sleeping sleep(), usleep() anywhere Only in process context. NEVER in interrupt context, under spinlock, or in atomic sections.
Pointers All pointers are your virtual addresses User pointers are DANGEROUS. Never dereference directly. Use copy_from_user()/copy_to_user().
Concurrency Threads + pthreads, userspace mutexes Spinlocks + mutexes + RCU + atomic ops. Multiple CPUs + preemption + interrupts = all can run your code simultaneously.

Day 1 Knowledge Check — Can You Answer These?

❓ Conceptual Questions

  • What physically prevents a user-space program from reading kernel memory? (Answer: page tables with supervisor bit + MMU)
  • Why can't you call printf() in a kernel driver? (Answer: No libc in kernel space, glibc isn't loaded)
  • What is the difference between a kernel built-in (obj-y) and a module (obj-m)? (Answer: built-in linked into vmlinux image, module is a separate .ko file loaded at runtime)
  • If your driver causes a NULL pointer dereference, what happens? (Answer: Kernel oops — CPU gets a page fault exception at address 0x0, kernel prints a backtrace to dmesg, may panic if panic_on_oops=1)
  • What instruction switches the CPU from Ring 3 to Ring 0 during a syscall? (Answer: SYSCALL instruction on x86-64; SVC on ARM64)
  • Can you use malloc() in a kernel driver? (Answer: NO. Use kmalloc(). malloc() is a glibc function that internally calls brk()/mmap() syscalls — which don't exist in kernel space.)

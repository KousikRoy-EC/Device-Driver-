The Problem — Why Ordinary Variables Break on SMP

💥 Three Enemies of Correct Concurrent Code

Before writing a single line of atomic code, you need to deeply understand why plain C variables are unsafe under concurrency. There are three distinct attackers, each requiring a different defence.

ENEMY 1: The Compiler Reorder ────────────────────────────────────────────────────────────────────────── Source code: What the compiler may emit: ───────────── ─────────────────────────── ready = 0; data = 42; ← reordered! data = 42; ready = 1; ← order swapped ready = 1; The compiler sees no dependency between 'data' and 'ready', so it is free to reorder them for register allocation or pipeline efficiency. If another CPU checks (ready == 1) and then reads data, it may see data = 0 even though we "assigned data before ready". ENEMY 2: The CPU Out-of-Order Execution ────────────────────────────────────────────────────────────────────────── Even if the compiler emits stores in order, modern CPUs have: - Store buffers: a write may sit in a buffer before reaching cache - Load reordering: a CPU may speculatively read a stale cache line - Write combining: adjacent writes may be merged and reordered CPU0 (writer) CPU1 (reader) ────────────── ────────────── data = 42 while (!ready) {} ← sees ready=1 ready = 1 use(data) ← may STILL see data=0! (CPU1's store buffer hasn't flushed 'data' to coherent cache yet on CPU0's side) ENEMY 3: Torn Reads/Writes (Non-Atomicity) ────────────────────────────────────────────────────────────────────────── On a 32-bit bus, writing a 64-bit counter requires two bus cycles: write high 32 bits → (interrupt fires here) → write low 32 bits A concurrent reader may see: new high bits + old low bits = garbage value. Even a single aligned 32-bit write is not guaranteed atomic at the C level — the compiler might generate load-modify-store as multiple instructions: LOAD r1, [counter] ADD r1, 1 ← another CPU can interrupt between these STORE [counter], r1

🛡️ The Defence Hierarchy

  • Locks (mutex/spinlock) — serialise access entirely. Correct but potentially slow. Use when the critical section is complex.
  • Atomic variables (atomic_t) — hardware-guaranteed RMW. For simple counters and flags. Faster than locks for single-variable updates.
  • READ_ONCE/WRITE_ONCE — prevent compiler reordering and tearing without hardware barriers. Cheapest — zero runtime cost.
  • Memory barriers — enforce ordering between memory operations across CPUs. Use to build lock-free data structures correctly.

📐 The Golden Rule

Every shared variable needs a synchronisation mechanism. The choice is cost vs complexity:

  • Simple counters — atomic_t (lock-free, cheap)
  • Flags/state — atomic_t or READ/WRITE_ONCE
  • Producer-consumer — barriers (ACQUIRE/RELEASE)
  • Complex structures — locks
  • Hot path, many CPUs — per-CPU variables

There is no correct shared variable without synchronisation. The compiler's strict aliasing and C11 memory model treats unsynchronised shared access as undefined behaviour.

🚨
The Volatile LieMany beginners add volatile to shared variables thinking it solves concurrency. It does NOT. volatile only prevents the compiler from caching a variable in a register across accesses — it does nothing about CPU reordering, cache coherence, or atomicity. The kernel explicitly forbids volatile for concurrency purposes. Use READ_ONCE()/WRITE_ONCE() instead — they do what you meant.

atomic_t and atomic64_t — Lock-Free Integer Operations

⚛️ Hardware-Guaranteed Read-Modify-Write

atomic_t is a kernel type (wrapping an int) whose operations are guaranteed to execute as a single indivisible instruction on the hardware — no other CPU can observe an intermediate state. On x86 this uses the LOCK prefix; on ARM it uses LDXR/STXR exclusive load/store pairs. The result: you get RMW operations without taking a lock.

atomic64_t wraps a 64-bit value. Use it for counters that can exceed 2³² (byte counts, timestamps). On 32-bit architectures this may require a spinlock internally, but the API is identical.

C — atomic_t — Complete API Reference
#include <linux/atomic.h>

/* ── Declaration and initialisation ──────────────────────────────── */
static atomic_t   counter   = ATOMIC_INIT(0);   /* static initialiser */
static atomic64_t bytes_out = ATOMIC64_INIT(0); /* 64-bit */

atomic_t runtime_counter;
atomic_set(&runtime_counter, 42);             /* runtime initialise */

/* ── Basic read and write ─────────────────────────────────────────── */
int val = atomic_read(&counter);               /* atomic load */
atomic_set(&counter, 100);                     /* atomic store */

/* ── Increment / Decrement ────────────────────────────────────────── */
atomic_inc(&counter);                           /* counter++  (no return) */
atomic_dec(&counter);                           /* counter--  (no return) */

/* Variants that RETURN the new value after the operation */
int new_val = atomic_inc_return(&counter);     /* ++counter */
int new_val = atomic_dec_return(&counter);     /* --counter */

/* Variants that RETURN the OLD value before the operation */
int old = atomic_fetch_inc(&counter);          /* old = counter; counter++ */
int old = atomic_fetch_dec(&counter);

/* Decrement and test for zero — very common pattern */
if (atomic_dec_and_test(&counter)) {           /* true if result == 0 */
    pr_info("last reference dropped\n");        /* used in refcounting! */
}

/* ── Add / Subtract ──────────────────────────────────────────────── */
atomic_add(5, &counter);                         /* counter += 5  (no return) */
atomic_sub(3, &counter);                         /* counter -= 3  (no return) */

int result = atomic_add_return(10, &counter);  /* counter += 10; return new */
int result = atomic_sub_return(2,  &counter);  /* counter -= 2;  return new */
int old    = atomic_fetch_add(10, &counter);   /* return old, then += 10 */

/* Subtract and test for zero or negative: */
if (atomic_sub_and_test(5, &counter))           /* true if result == 0 */
    pr_info("counter hit zero\n");

/* ── Bitwise operations ──────────────────────────────────────────── */
atomic_or  (BIT(3), &counter);                  /* counter |= BIT(3) */
atomic_and (0xFF, &counter);                     /* counter &= 0xFF  */
atomic_xor (BIT(0), &counter);                  /* counter ^= BIT(0) */
int old = atomic_fetch_or(BIT(5), &counter);   /* returns old value */
int old = atomic_fetch_and(0x0F, &counter);

/* ── atomic64_t — same API with 64_ prefix ───────────────────────── */
atomic64_inc(&bytes_out);
atomic64_add(n_written, &bytes_out);
u64 total = atomic64_read(&bytes_out);

/* ── Real-world usage patterns ───────────────────────────────────── */
/* Pattern 1: Reference counting (replace kref for simple cases) */
struct my_obj { atomic_t refcount; ... };
atomic_inc(&obj->refcount);
if (atomic_dec_and_test(&obj->refcount))
    kfree(obj);   /* last reference — free it */

/* Pattern 2: Open file counter (limit concurrent opens) */
static atomic_t open_count = ATOMIC_INIT(0);
static int mydev_open(struct inode *i, struct file *f)
{
    if (atomic_inc_return(&open_count) > 4) {
        atomic_dec(&open_count);
        return -EBUSY;   /* max 4 simultaneous opens */
    }
    return 0;
}
static int mydev_release(struct inode *i, struct file *f)
{
    atomic_dec(&open_count);
    return 0;
}

/* Pattern 3: Error counter (increment on error, never reset) */
static atomic64_t err_count = ATOMIC64_INIT(0);
if (hw_error)
    atomic64_inc(&err_count);
/* Safe to read from /proc show without locking */
seq_printf(m, "errors: %lld\n", atomic64_read(&err_count));
📌
atomic_t Ordering GuaranteesThe basic atomic_inc, atomic_add, etc. operations on x86 are fully ordered because x86's strong memory model includes an implicit barrier. On ARM/RISC-V (weak memory model), basic atomics use LDADD/STADD which are also fully ordered. However, for the release/acquire pattern (producer-consumer), use the explicit smp_store_release/smp_load_acquire variants covered in Section 5 — they allow the compiler to emit lighter barriers on architectures that support it.

cmpxchg, xchg, and CAS — Building Lock-Free Algorithms

🔄 Compare-And-Swap: The Mother of All Lock-Free Primitives

Compare-and-swap (CAS) is the fundamental primitive from which virtually all lock-free data structures are built. It atomically does: "if the memory location contains the value I expect, replace it with a new value — otherwise tell me what was actually there." This lets you attempt an update and retry if another CPU raced with you — all without a lock.

On x86 this compiles to a single CMPXCHG instruction (with LOCK prefix for multi-core). On ARM it uses a LDXR/STXR loop.

C — cmpxchg, xchg, xadd — Complete Reference
#include <linux/atomic.h>   /* cmpxchg, xchg */
#include <asm/cmpxchg.h>    /* architecture-specific implementation */

/* ── cmpxchg: Compare and Swap ───────────────────────────────────────
 *
 * Signature: TYPE cmpxchg(TYPE *ptr, TYPE old, TYPE new)
 *
 * Semantics (atomically):
 *   if (*ptr == old) { *ptr = new; }
 *   return previous_value_of(*ptr);
 *
 * Returns: the VALUE that was in *ptr before the operation.
 * Success: returned value == old  (the swap happened)
 * Failure: returned value != old  (another CPU changed it first)
 * ─────────────────────────────────────────────────────────────────── */

/* Example 1: Atomic state machine transition */
#define STATE_IDLE    0
#define STATE_RUNNING 1
#define STATE_STOPPED 2

static atomic_t dev_state = ATOMIC_INIT(STATE_IDLE);

bool try_start_device(void)
{
    int prev = atomic_cmpxchg(&dev_state, STATE_IDLE, STATE_RUNNING);
    /* Succeeds ONLY if state was IDLE — exactly one caller wins */
    return prev == STATE_IDLE;
}

/* Example 2: Lock-free maximum — update max if our value is larger */
static atomic_t peak_value = ATOMIC_INIT(0);

void update_peak(int new_val)
{
    int old, current;
    do {
        old = atomic_read(&peak_value);
        if (new_val <= old) return;   /* already at or above new value */
        current = atomic_cmpxchg(&peak_value, old, new_val);
    } while (current != old);         /* retry if another CPU won the race */
}

/* Example 3: Once-only initialisation flag */
static atomic_t initialised = ATOMIC_INIT(0);

void maybe_init(void)
{
    /* Only the ONE caller that successfully changes 0→1 runs init */
    if (atomic_cmpxchg(&initialised, 0, 1) == 0)
        do_expensive_init();
}

/* ── cmpxchg on raw pointers (not atomic_t) ─────────────────────────
 * Works on any naturally-aligned pointer-sized or smaller type.
 * Often used for pointer swapping in lock-free lists/queues.  */
void *old_ptr = cmpxchg(&list->head, expected_ptr, new_ptr);
if (old_ptr == expected_ptr)
    pr_debug("pointer swapped atomically\n");

/* ── xchg: Unconditional swap — returns old value ───────────────────
 * Atomically: old = *ptr; *ptr = new; return old;
 * Use when you always want to replace (no conditional check needed).  */
int prev = xchg(&dev->flags, NEW_FLAGS);
void *old_buf = xchg(&dev->buf_ptr, new_buf);

/* ── atomic_xchg: xchg on atomic_t ─────────────────────────────────*/
int old_state = atomic_xchg(&dev_state, STATE_STOPPED);

/* ── try_cmpxchg: cleaner API (kernel 5.x+) ─────────────────────────
 * Returns bool: true = swap happened, false = failed.
 * *oldp is updated with current value on failure.  */
int expected = STATE_IDLE;
if (atomic_try_cmpxchg(&dev_state, &expected, STATE_RUNNING))
    pr_info("started — was IDLE\n");
else
    pr_info("failed — state was %d, not IDLE\n", expected);

/* ── cmpxchg64: explicit 64-bit CAS ─────────────────────────────────
 * Use on 32-bit platforms where 64-bit CAS is explicit.
 * On 64-bit platforms cmpxchg() handles 64-bit naturally.  */
u64 old64 = cmpxchg64(&dev->timestamp, old_ts, new_ts);

⚠️ The ABA Problem — A Trap in Lock-Free Code

CAS has a subtle failure mode: A→B→A. Thread 1 reads value A, gets preempted. Thread 2 changes A→B→A. Thread 1 resumes, CAS succeeds (sees A again) — but the world has changed underneath it. In simple integer counters the ABA problem usually doesn't matter, but in pointer-based lock-free data structures it can cause use-after-free bugs. Solutions include: version/stamp counters (pack a version into the high bits of a pointer), hazard pointers, or just using RCU (Day 11's topic).

READ_ONCE / WRITE_ONCE — Taming the Compiler

🔍 The Cheapest Synchronisation Primitive

READ_ONCE(x) and WRITE_ONCE(x, val) are pure compile-time constructs — zero hardware instructions, zero runtime cost. They tell the compiler two things: (1) treat this access as volatile (don't cache it in a register, don't eliminate it, don't merge it), and (2) the access must be done in a single instruction (no tearing). They prevent compiler reordering and non-atomic compilation of the access itself, but they do not generate hardware memory barriers — other CPUs can still see a different order. For that you need the barriers in Section 5.

C — READ_ONCE / WRITE_ONCE — When and How
#include <linux/compiler.h>  /* READ_ONCE, WRITE_ONCE, barrier() */

/* ── WITHOUT READ_ONCE — the compiler may do this: ──────────────────
 *
 * Source:                    Compiler may emit:
 *   if (dev->running)          r1 = dev->running
 *     do_work();               if (!r1) goto skip
 *   if (dev->running)          if (!r1) goto skip   ← uses cached r1!
 *     do_more_work();          do_work(); do_more_work();
 *                          skip:
 *
 * The compiler sees no write to dev->running in between, so it reuses
 * the first read. But another CPU may have cleared it between the two
 * checks. This is the "load fusing" optimisation — dangerous for shared
 * state flags.
 * ─────────────────────────────────────────────────────────────────── */

/* WITH READ_ONCE — each access is a fresh load from memory */
if (READ_ONCE(dev->running))     /* load 1: always reads from memory */
    do_work();
if (READ_ONCE(dev->running))     /* load 2: fresh load, sees any CPU write */
    do_more_work();

/* ── WRITE_ONCE — prevent store tearing and merging ──────────────── */
/* Without WRITE_ONCE, compiler may split a store into multiple ops,
 * or reorder it with adjacent stores it sees as independent. */
WRITE_ONCE(dev->running, false); /* single, non-torn, non-reordered store */

/* ── Classic producer-consumer flag with barriers ────────────────── */
/* (This is the CORRECT pattern — barriers are covered in Section 5) */

/* Producer: */
WRITE_ONCE(shared_buf[idx], data);     /* write data first */
smp_wmb();                              /* barrier: data visible before ready */
WRITE_ONCE(ready, true);               /* then set ready flag */

/* Consumer: */
while (!READ_ONCE(ready)) cpu_relax(); /* spin until ready */
smp_rmb();                              /* barrier: see data written before ready */
int val = READ_ONCE(shared_buf[idx]);  /* now safe to read */

/* ── barrier(): compiler-only barrier, no CPU instruction ───────────
 * Prevents the compiler from moving any memory access across this point.
 * Does NOT prevent CPU reordering. Use for:
 *   - Preventing speculative reads past a check
 *   - Ensuring a volatile-like sequence in generated code             */
barrier();

/* ── ACCESS_ONCE (deprecated, use READ_ONCE) ─────────────────────── */
/* You may see ACCESS_ONCE() in old code — it's the same as READ_ONCE. */

/* ── When to use READ_ONCE vs atomic_t ──────────────────────────────
 *
 * READ_ONCE + WRITE_ONCE:
 *   Use when:  the access itself won't tear (native-width aligned)
 *              AND you only need compiler ordering (not CPU ordering)
 *   Typical:   bool flags, pointer reads, enum state fields
 *              (combined with explicit smp_* barriers)
 *
 * atomic_t:
 *   Use when:  you need RMW operations (inc, dec, add, CAS)
 *              OR you need the atomic guarantee across architectures
 *   Typical:   counters, refcounts, open_count, state with transitions
 * ─────────────────────────────────────────────────────────────────── */

Memory Barriers — Enforcing Order Across CPUs

🧱 The CPU Memory Model — What Reordering Can Happen

Every CPU architecture has a memory model — a specification of which store/load reorderings the hardware is allowed to perform. x86 has a very strong model (almost no reordering). ARM, RISC-V, and Power have weak models (significant reordering). The Linux kernel must run correctly on all of them, so it uses explicit memory barriers to express ordering constraints in architecture-independent code.

CPU MEMORY REORDERING POSSIBILITIES (weak model, e.g. ARM): ────────────────────────────────────────────────────────────────────────────── These 4 reorderings can all happen on a weak-memory architecture: LoadLoad: A=READ(x); B=READ(y) → CPU may do B=READ(y) before A=READ(x) LoadStore: A=READ(x); WRITE(y,1) → CPU may do WRITE(y,1) before A=READ(x) StoreLoad: WRITE(x,1); B=READ(y) → CPU may do B=READ(y) before WRITE(x,1) StoreStore: WRITE(x,1); WRITE(y,2)→ CPU may do WRITE(y,2) before WRITE(x,1) x86 ONLY allows StoreLoad reordering (TSO = Total Store Order). ARM/RISC-V/Power allow ALL FOUR. The kernel's smp_* barriers block them. ────────────────────────────────────────────────────────────────────────────── BARRIER TYPES AND WHAT THEY BLOCK: smp_rmb() Read Memory Barrier — blocks LoadLoad reordering "All reads before this barrier complete before any reads after" smp_wmb() Write Memory Barrier — blocks StoreStore reordering "All writes before this barrier are visible before any after" smp_mb() Full Memory Barrier — blocks ALL four reorderings "All reads+writes before this barrier happen before any after" Most expensive. Use only when needed. smp_store_release(ptr, val) Write + acquire-like barrier (see below) smp_load_acquire(ptr) Read + release-like barrier (see below)
C — smp_mb / smp_rmb / smp_wmb — When to Use Each
#include <asm/barrier.h>   /* smp_mb, smp_rmb, smp_wmb — included via linux/smp.h */

/* ─────────────────────────────────────────────────────────────────────
 * PATTERN 1: smp_wmb() + smp_rmb() — Producer-Consumer
 *
 * Use when one CPU writes data then signals "ready",
 * another CPU checks "ready" then reads data.
 * ───────────────────────────────────────────────────────────────────── */

/* CPU0 — Producer */
msg->data    = 42;
msg->payload = ptr;
smp_wmb();            /* ALL stores above MUST be visible before... */
WRITE_ONCE(msg->ready, true);   /* ...this store becomes visible */

/* CPU1 — Consumer */
while (!READ_ONCE(msg->ready))
    cpu_relax();
smp_rmb();            /* ALL loads below MUST happen after... */
int   data = msg->data;      /* ...the ready load above completes */
void *p    = msg->payload;   /* guaranteed to see the producer's values */

/* ─────────────────────────────────────────────────────────────────────
 * PATTERN 2: smp_mb() — Full Barrier for Complex Ordering
 *
 * Use when you need to ensure that ALL prior reads AND writes are
 * visible before continuing. More expensive than rmb/wmb.
 * ───────────────────────────────────────────────────────────────────── */

/* Classic lock-free flag race prevention */
WRITE_ONCE(need_resched, true);
smp_mb();                     /* full barrier: write above, read below */
if (READ_ONCE(thread_sleeping))
    wake_up(&wq);

/* Without smp_mb(), the thread might do:
 *   CPU0 (setter):    CPU1 (thread):
 *   need_resched=1    if (!need_resched) goto sleep  ← reads 0 (old)
 *   read sleeping=0   sleeping=1
 * → thread goes to sleep, setter saw sleeping=0 → never woken up! */

/* ─────────────────────────────────────────────────────────────────────
 * PATTERN 3: smp_store_release + smp_load_acquire — Modern Idiom
 *
 * The PREFERRED modern way to write producer-consumer code.
 * Equivalent to wmb/rmb pair but expressed at the OPERATION level,
 * which allows the compiler to generate optimal code per-architecture.
 *
 * smp_store_release(ptr, val):
 *   All prior loads/stores happen BEFORE this store. (release semantics)
 *
 * smp_load_acquire(ptr):
 *   All subsequent loads/stores happen AFTER this load. (acquire semantics)
 * ───────────────────────────────────────────────────────────────────── */

/* CPU0 — Producer (release) */
ring->buf[tail] = item;
smp_store_release(&ring->tail, (tail + 1) & RING_MASK);
/* item write GUARANTEED visible before tail update */

/* CPU1 — Consumer (acquire) */
unsigned int head = smp_load_acquire(&ring->tail);
int item = ring->buf[head];  /* GUARANTEED to see producer's item write */

/* ─────────────────────────────────────────────────────────────────────
 * PATTERN 4: dma_wmb() / dma_rmb() — DMA Coherency Barriers
 *
 * For communication between CPU and DMA hardware (not between CPUs).
 * Used in NIC drivers, storage drivers, GPU drivers.
 * ───────────────────────────────────────────────────────────────────── */

/* Set up DMA descriptor then tell hardware it's ready */
desc->addr   = dma_addr;
desc->length = len;
desc->flags  = DESC_VALID;
dma_wmb();                /* ensure descriptor fields written before owner bit */
desc->owner  = HW_OWNED;  /* hardware starts DMA */

/* After hardware returns descriptor, read results */
while (desc->owner != CPU_OWNED) cpu_relax();
dma_rmb();                /* ensure owner read completes before we read results */
int status = desc->status;

Per-CPU Variables — Lock-Free by Design

⚡ The Fastest Path: Eliminate Sharing Entirely

The fastest way to avoid a race condition is to ensure each CPU operates on its own copy of a variable, with no sharing at all. This is what per-CPU variables do. The kernel allocates a separate instance of the variable for each CPU in the system. As long as you disable preemption while accessing it (so you can't migrate to another CPU mid-operation), there are no races — no locks, no barriers needed.

Per-CPU variables are ideal for high-frequency per-CPU statistics (packet counters, cache miss counters, scheduler runqueue depth) that are summed up occasionally. They're how the kernel implements things like /proc/stat CPU counters with zero contention across hundreds of cores.

C — Per-CPU Variables — Declaration, Access, and Aggregation
#include <linux/percpu.h>
#include <linux/percpu-defs.h>

/* ── Static per-CPU variable (declared at file scope) ────────────── */
DEFINE_PER_CPU(unsigned long, pkt_count);    /* one ulong per CPU */
DEFINE_PER_CPU(struct stats, cpu_stats);     /* one struct per CPU */
DEFINE_PER_CPU(int, irq_count) = 0;          /* with initialiser */

/* ── Dynamic per-CPU allocation (for module use) ─────────────────── */
unsigned long __percpu *dyn_counter;
dyn_counter = alloc_percpu(unsigned long);
if (!dyn_counter) return -ENOMEM;
/* Cleanup: */
free_percpu(dyn_counter);

/* ── Accessing per-CPU variables — CRITICAL: disable preemption ──────
 *
 * You must prevent the scheduler from migrating you to another CPU
 * between the get_cpu_var() and put_cpu_var() calls.
 *
 * get_cpu_var(var):   disables preemption, returns reference to THIS cpu's var
 * put_cpu_var(var):   re-enables preemption
 * ─────────────────────────────────────────────────────────────────── */

/* Pattern 1: get_cpu_var / put_cpu_var (disables preemption) */
get_cpu_var(pkt_count)++;                     /* increment this CPU's counter */
put_cpu_var(pkt_count);                       /* restore preemption */

/* For structs: */
struct stats *s = &get_cpu_var(cpu_stats);
s->rx_packets++;
s->rx_bytes += len;
put_cpu_var(cpu_stats);

/* Pattern 2: this_cpu_* — most efficient, assumes preemption already off
 * (Inside IRQ handler, spinlock held, or after get_cpu()) */
this_cpu_inc(pkt_count);                      /* faster: uses per-cpu ops directly */
this_cpu_add(pkt_count, n);
this_cpu_write(pkt_count, 0);
unsigned long n = this_cpu_read(pkt_count);

/* Pattern 3: per_cpu_ptr — access another CPU's copy */
int cpu;
for_each_possible_cpu(cpu) {
    unsigned long *p = per_cpu_ptr(&pkt_count, cpu);
    pr_info("CPU%d: %lu packets\n", cpu, *p);
}

/* ── Aggregating per-CPU counters (for /proc stats) ──────────────── */
unsigned long get_total_packets(void)
{
    unsigned long total = 0;
    int cpu;
    for_each_possible_cpu(cpu)
        total += per_cpu(pkt_count, cpu);   /* per_cpu(var, cpu) = direct access */
    return total;
}

/* ── In an IRQ handler — per-CPU is ideal here ───────────────────── */
static irqreturn_t mydrv_irq(int irq, void *dev_id)
{
    /* IRQ context: preemption is disabled, safe to use this_cpu_* */
    this_cpu_inc(irq_count);           /* no lock, no barrier, no contention */
    this_cpu_add(pkt_count, hw_count());
    return IRQ_HANDLED;
}

/* ── DECLARE_PER_CPU for extern declarations (header files) ──────── */
DECLARE_PER_CPU(unsigned long, pkt_count);   /* in .h */
DEFINE_PER_CPU(unsigned long, pkt_count);    /* in .c */

Atomic Bit Operations — set_bit, clear_bit, test_and_set_bit

🔢 Lock-Free Flag Bitmaps

The kernel provides a family of atomic bit operations that work directly on unsigned long bitmaps — no need for a separate mutex to protect a flags field. These are used everywhere: network device flags (NETIF_F_*), driver state bitmaps, page flags, task state flags. On x86 they use the LOCK BTS/BTR/BTC instructions; on ARM they use exclusive load-store pairs.

C — Atomic Bit Operations — The Full API
#include <linux/bitops.h>

unsigned long flags = 0;    /* a bitmap — can be any size */

/* ── Basic atomic bit operations ─────────────────────────────────────
 * These are ATOMIC — safe to call without locks on shared bitmaps.
 * nr = bit number (0 = LSB). addr = pointer to unsigned long bitmap. */

set_bit  (3, &flags);   /* atomically set   bit 3 (flags |= BIT(3)) */
clear_bit(3, &flags);   /* atomically clear bit 3 (flags &= ~BIT(3)) */
change_bit(3,&flags);   /* atomically toggle bit 3 (flags ^= BIT(3)) */
int val = test_bit(3, &flags); /* read bit 3 (non-atomic read, see note) */

/* ── Test-and-set / Test-and-clear — atomically operate AND return old */
int was_set   = test_and_set_bit  (5, &flags); /* set, return old value */
int was_clear = test_and_clear_bit(5, &flags); /* clear, return old */
int was_set   = test_and_change_bit(5,&flags); /* toggle, return old */

/* ── Non-atomic variants (use only when you hold a lock or are UP) ── */
__set_bit  (3, &flags);   /* NOT atomic — faster, for locked sections */
__clear_bit(3, &flags);
__change_bit(3,&flags);
int old = __test_and_set_bit(3, &flags);   /* non-atomic TAS */

/* ── Driver state machine with bit flags ─────────────────────────── */
#define DEV_OPEN       0
#define DEV_ENABLED    1
#define DEV_IRQ_ACTIVE 2
#define DEV_RESETTING  3

struct mydev { unsigned long state_flags; ... };

/* Open: set OPEN flag, fail if already open */
if (test_and_set_bit(DEV_OPEN, &dev->state_flags))
    return -EBUSY;    /* bit was already set — device already open */

/* Release: clear OPEN flag */
clear_bit(DEV_OPEN, &dev->state_flags);

/* Check multiple flags in IRQ handler */
if (test_bit(DEV_RESETTING, &dev->state_flags))
    return IRQ_HANDLED;   /* ignore IRQs during reset */

/* ── find_first_bit, for_each_set_bit — traverse bitmaps ─────────── */
unsigned long active = 0xB5;      /* 10110101 in binary */
int bit = find_first_bit(&active, 8);   /* = 0 */
int bit = find_next_bit (&active, 8, 1);  /* = 2 (next after bit 1) */

unsigned long alert_map;
int nr;
for_each_set_bit(nr, &alert_map, BITS_PER_LONG)
    handle_alert(nr);   /* called once per set bit */

Practice Project — Lock-Free Statistics Engine for pipdev

🎯
Today's GoalReplace all the mutex-protected stat counters in pipdev with a lock-free statistics engine using the primitives from today: atomic64_t for byte/call counters, per-CPU variables for per-call hot-path accounting, atomic_t with cmpxchg for peak tracking, set_bit/clear_bit for a state bitmap, and smp_store_release/smp_load_acquire for the ring-buffer producer-consumer between write() and read().
C — pipdev_atomic.c — Lock-Free Statistics Engine
/*
 * pipdev_atomic.c — Day 12: Replace mutex-protected stats with atomics
 *
 * Demonstrates every primitive from today in a realistic driver context.
 */
#include <linux/atomic.h>
#include <linux/percpu.h>
#include <linux/bitops.h>

/* ── State bitmap — replaces the bool fields ─────────────────────── */
#define PIPE_OPEN       0   /* device has open file descriptors */
#define PIPE_FULL       1   /* kfifo is full */
#define PIPE_DROP_MODE  2   /* drop-on-full enabled (sysfs config) */
#define PIPE_DRAINING   3   /* flush in progress */

/* ── Atomic statistics counters ─────────────────────────────────────*/
struct pipdev_stats {
    atomic64_t  total_reads;       /* total read() calls */
    atomic64_t  total_writes;      /* total write() calls */
    atomic64_t  total_rx_bytes;    /* bytes given to readers */
    atomic64_t  total_tx_bytes;    /* bytes received from writers */
    atomic64_t  total_dropped;     /* bytes dropped (drop_on_full mode) */
    atomic_t    peak_write_size;   /* largest single write */
    atomic_t    concurrent_opens;  /* current open fd count */
    unsigned long state;           /* PIPE_* flag bits (bitops-protected) */
};

/* ── Per-CPU hot-path counters (zero contention) ─────────────────── */
DEFINE_PER_CPU(unsigned long, cpu_reads);
DEFINE_PER_CPU(unsigned long, cpu_writes);
DEFINE_PER_CPU(unsigned long, cpu_bytes_rx);
DEFINE_PER_CPU(unsigned long, cpu_bytes_tx);

/* ── Extended pipdev struct ──────────────────────────────────────── */
struct pipdev {
    struct kfifo          fifo;
    struct mutex          lock;          /* only for fifo access */
    wait_queue_head_t     read_wq;
    wait_queue_head_t     write_wq;
    struct pipdev_stats   stats;
};

/* ── Stats initialisation ────────────────────────────────────────── */
static void stats_init(struct pipdev_stats *s)
{
    atomic64_set(&s->total_reads,     0);
    atomic64_set(&s->total_writes,    0);
    atomic64_set(&s->total_rx_bytes,  0);
    atomic64_set(&s->total_tx_bytes,  0);
    atomic64_set(&s->total_dropped,   0);
    atomic_set (&s->peak_write_size,  0);
    atomic_set (&s->concurrent_opens, 0);
    s->state = 0;
}

/* ── open() — atomic open-count with limit enforcement ───────────── */
static int pipdev_open(struct inode *inode, struct file *filp)
{
    struct pipdev *dev = ((struct miscdevice *)inode->i_cdev)->this_device->driver_data;
    filp->private_data = dev;

    if (atomic_inc_return(&dev->stats.concurrent_opens) > 16) {
        atomic_dec(&dev->stats.concurrent_opens);
        return -EBUSY;
    }
    set_bit(PIPE_OPEN, &dev->stats.state);
    return nonseekable_open(inode, filp);
}

/* ── release() — decrement open count ────────────────────────────── */
static int pipdev_release(struct inode *inode, struct file *filp)
{
    struct pipdev *dev = filp->private_data;
    if (atomic_dec_and_test(&dev->stats.concurrent_opens))
        clear_bit(PIPE_OPEN, &dev->stats.state);
    return 0;
}

/* ── write() — per-CPU hot path, peak tracking via cmpxchg ──────── */
static ssize_t pipdev_write(struct file *filp, const char __user *ubuf,
                              size_t count, loff_t *ppos)
{
    struct pipdev *dev = filp->private_data;
    unsigned int copied; int ret;

    if (mutex_lock_interruptible(&dev->lock)) return -EINTR;
    ret = kfifo_from_user(&dev->fifo, ubuf, count, &copied);
    mutex_unlock(&dev->lock);
    if (ret) return -EFAULT;

    /* Per-CPU counters — hot path, no contention */
    this_cpu_inc(cpu_writes);
    this_cpu_add(cpu_bytes_tx, copied);

    /* Global atomic stats (slightly slower, but accurate totals) */
    atomic64_inc(&dev->stats.total_writes);
    atomic64_add(copied, &dev->stats.total_tx_bytes);

    /* Track peak write size via CAS loop */
    int old, cur = (int)count;
    do {
        old = atomic_read(&dev->stats.peak_write_size);
        if (cur <= old) break;
    } while (atomic_cmpxchg(&dev->stats.peak_write_size, old, cur) != old);

    /* Update state bit */
    if (kfifo_is_full(&dev->fifo))
        set_bit(PIPE_FULL, &dev->stats.state);
    else
        clear_bit(PIPE_FULL, &dev->stats.state);

    wake_up_interruptible(&dev->read_wq);
    return copied;
}

/* ── /proc show — aggregate per-CPU + global atomics ─────────────── */
static int pipdev_proc_show(struct seq_file *m, void *v)
{
    struct pipdev *dev = m->private;
    struct pipdev_stats *s = &dev->stats;

    /* Sum per-CPU write counters */
    unsigned long cpu_w = 0, cpu_b = 0;
    int cpu;
    for_each_possible_cpu(cpu) {
        cpu_w += per_cpu(cpu_writes,   cpu);
        cpu_b += per_cpu(cpu_bytes_tx, cpu);
    }

    seq_puts(m, "=== pipdev statistics ===\n");
    seq_printf(m, "State flags:    0x%lx  [%s%s%s%s]\n",
               s->state,
               test_bit(PIPE_OPEN,      &s->state) ? "OPEN " : "",
               test_bit(PIPE_FULL,      &s->state) ? "FULL " : "",
               test_bit(PIPE_DROP_MODE, &s->state) ? "DROP " : "",
               test_bit(PIPE_DRAINING,  &s->state) ? "DRAIN" : "");
    seq_printf(m, "Open handles:   %d\n",   atomic_read(&s->concurrent_opens));
    seq_printf(m, "Total reads:    %lld\n",  atomic64_read(&s->total_reads));
    seq_printf(m, "Total writes:   %lld  (per-cpu sum: %lu)\n",
               atomic64_read(&s->total_writes), cpu_w);
    seq_printf(m, "TX bytes:       %lld  (per-cpu sum: %lu)\n",
               atomic64_read(&s->total_tx_bytes), cpu_b);
    seq_printf(m, "Peak write:     %d bytes\n", atomic_read(&s->peak_write_size));
    seq_printf(m, "Dropped:        %lld bytes\n", atomic64_read(&s->total_dropped));
    return 0;
}

Common Bugs & How to Detect Them

🐛 Bug: Using atomic_t to guard a struct

atomic_t makes individual integer operations atomic, but it does not make a struct thread-safe. If you have a struct with 5 fields and you protect it with an atomic "lock flag", you still have a race — another CPU can read 3 fields while you update the other 2. Use a real mutex for struct-level invariants. Atomics are for scalar values only.

BUG / FIX
/* BUG: atomic "lock" protects nothing */
if (atomic_cmpxchg(&lock, 0, 1) == 0) {
    dev->x = a;   /* NOT protected! */
    dev->y = b;   /* race window here */
    atomic_set(&lock, 0);
}
/* FIX: use a real spinlock */
spin_lock(&dev->lock);
dev->x = a; dev->y = b;
spin_unlock(&dev->lock);

🐛 Bug: Missing barrier in flag protocol

Writing the "ready" flag before the data is visible to other CPUs — the classic producer-consumer bug. Works on x86 (strong model) but silently fails on ARM. KCSAN (Kernel Concurrency Sanitizer) can catch this.

BUG / FIX
/* BUG: no barrier — ARM may reorder */
dev->data  = value;
dev->ready = true;    /* may be visible BEFORE data! */
/* FIX: release semantics */
dev->data = value;
smp_store_release(&dev->ready, true);

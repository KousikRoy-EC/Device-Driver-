Physical vs Virtual Memory — The Foundation

🧠 The Mental Model Every Driver Dev Must Have

There are two completely separate address spaces you will deal with as a driver developer. Confusing them is one of the most common and dangerous mistakes. Every address you see in kernel code is a virtual address — but the hardware (DMA engines, MMIO registers) operates on physical addresses.

PHYSICAL MEMORY (what RAM chips actually contain) ────────────────────────────────────────────────────────────────────────── 0x0000_0000_0000 ┌──────────────────┐ │ Reserved / BIOS │ (first 1MB on x86) 0x0000_0010_0000 ├──────────────────┤ │ │ │ Available RAM │ (your machine's actual memory sticks) │ (e.g., 8 GB) │ DMA engines use these addresses │ │ 0x0002_0000_0000 ├──────────────────┤ (8 GB boundary) │ MMIO Regions │ (PCIe BAR registers, GPU VRAM aperture) │ (no RAM here) │ read/write these to talk to hardware └──────────────────┘ VIRTUAL MEMORY (what the CPU's MMU presents to software) ────────────────────────────────────────────────────────────────────────── 0x0000_0000_0000_0000 ┌──────────────────────────────────────┐ │ USER SPACE (128 TB) │ │ Each process has its OWN mapping │ │ Process A and Process B can both │ │ have ptr=0x400000 → different RAM │ 0x0000_7FFF_FFFF_FFFF ├──────────────────────────────────────┤ │ (CANONICAL HOLE — unmapped) │ 0xFFFF_8000_0000_0000 ├──────────────────────────────────────┤ │ KERNEL SPACE (128 TB) │ │ │ 0xFFFF_8880_0000_0000│ ← PAGE_OFFSET │ │ Direct Map: ALL physical RAM │ │ mapped here, linearly │ │ phys 0x0 → virt PAGE_OFFSET+0x0 │ │ phys 0x1000 → virt PAGE_OFFSET+0x1000│ │ │ 0xFFFF_C900_0000_0000│ vmalloc area (vmalloc/ioremap) │ 0xFFFF_E000_0000_0000│ modules area (.ko loaded here) │ 0xFFFF_FFFF_8000_0000│ kernel code (_text, _data, _bss) │ 0xFFFF_FFFF_FFFF_FFFF └──────────────────────────────────────┘ KEY CONVERSION FUNCTIONS: virt_to_phys(vaddr) → physical address (only for direct-map addresses!) phys_to_virt(paddr) → virtual address (only for direct-mapped RAM) __pa(vaddr) → same as virt_to_phys (macro shorthand) __va(paddr) → same as phys_to_virt (macro shorthand)

🗺️ The Direct Map Region

Linux maps all physical RAM into kernel virtual space at a fixed offset (PAGE_OFFSET, typically 0xffff888000000000 on x86-64). This means every physical RAM byte has a kernel virtual address you can use directly.

kmalloc() returns addresses in this region — that's why virt_to_phys(kmalloc_ptr) gives you a valid physical address for DMA. vmalloc addresses are NOT in the direct map — never pass them to DMA!

⚠️ Why This Matters for Drivers

  • DMA needs physical addresses — use dma_map_single(), not raw pointers
  • MMIO is not RAM — use ioremap() to access hardware registers
  • Userspace pointers are virtual — different page table from kernel, never dereference directly
  • vmalloc != kmalloc — vmalloc is virtually contiguous but scattered in physical RAM, cannot use for DMA

Memory Zones — Where RAM is Categorized

🗂️ Why Zones Exist

Not all RAM is equal. Legacy hardware (ISA DMA controllers) can only address the first 16MB. 32-bit devices can only address up to 4GB. The kernel divides physical RAM into zones and lets allocators request from specific zones via GFP flags.

GFP Flags — The Most Important Parameter in Kernel Allocation

🚩 GFP = Get Free Pages — Allocation Behavior Flags

Every kernel allocation call takes a gfp_t flags argument. These flags control how the allocator behaves when memory is tight — can it sleep and wait? Can it trigger memory reclaim? Can it use emergency reserves? Getting this wrong either crashes your driver or deadlocks the kernel.

✅ GFP_KERNEL

  • Use in: Process context only
  • Can sleep: YES
  • Can trigger reclaim: YES
  • Can call OOM killer: YES
  • Default for drivers — use this in probe(), init(), ioctl() handlers. The allocator will wait for memory to become available.

⚡ GFP_ATOMIC

  • Use in: Interrupt context, spinlock held
  • Can sleep: NO — never
  • Can trigger reclaim: NO
  • Uses emergency reserves: YES
  • IRQ handlers, tasklets, timer callbacks. Will fail (return NULL) instead of waiting. Always check return value!

🔒 GFP_NOWAIT

  • Use in: Process context but can't sleep
  • Can sleep: NO
  • Can trigger reclaim: NO
  • Uses emergency reserves: NO
  • Like GFP_ATOMIC but doesn't touch reserves. More likely to fail. Rarely needed in drivers.

🧹 GFP_NOIO / GFP_NOFS

  • GFP_NOIO: Cannot start any I/O during reclaim
  • GFP_NOFS: Cannot call filesystem code
  • Used in: Storage/filesystem drivers
  • Prevents deadlocks where reclaim tries to do I/O which needs memory which triggers reclaim...
C — GFP Flag Composition — How Flags Combine
/* GFP flags are bit masks that combine zone + behavior flags.
 * GFP_KERNEL is actually a combination of multiple flags:
 *
 * #define GFP_KERNEL  (__GFP_RECLAIM | __GFP_IO | __GFP_FS)
 * #define GFP_ATOMIC  (__GFP_HIGH | __GFP_ATOMIC | __GFP_KSWAPD_RECLAIM)
 *
 * You can compose flags yourself for specific needs: */

/* Allocate from DMA zone, can sleep */
buf = kmalloc(size, GFP_KERNEL | GFP_DMA);

/* Allocate zeroed memory that can be used for DMA32 */
buf = kzalloc(size, GFP_KERNEL | GFP_DMA32);

/* Atomic allocation without touching emergency reserves */
buf = kmalloc(size, GFP_NOWAIT);

/* __GFP_ZERO: zero the memory (kzalloc is kmalloc + this flag) */
buf = kmalloc(size, GFP_KERNEL | __GFP_ZERO);
/* Equivalent to kzalloc(size, GFP_KERNEL) */

/* __GFP_NOFAIL: keep trying until successful (rarely appropriate) */
buf = kmalloc(size, GFP_KERNEL | __GFP_NOFAIL);
/* WARNING: This will loop forever if memory never becomes available.
 * Only use when you CANNOT handle allocation failure. Very rare. */

/* __GFP_ACCOUNT: charge to memory cgroup (for user-facing allocations) */
buf = kmalloc(size, GFP_KERNEL | __GFP_ACCOUNT);
🚨
The Cardinal Sin: GFP_KERNEL in Atomic ContextUsing GFP_KERNEL while holding a spinlock or in an interrupt handler causes a BUG: sleeping function called from invalid context kernel oops. The allocator tries to sleep, but sleeping is impossible. This is one of the most common bugs in new drivers. If you're in doubt whether you can sleep — call might_sleep() at the top of your function to get a warning during development.

kmalloc / kzalloc — The SLUB Allocator

🔧 kmalloc — Your Primary Allocation Tool

kmalloc() is backed by the SLUB allocator (Slab allocator with fewer queues). SLUB maintains pre-allocated caches of fixed-size objects (32B, 64B, 128B, 256B... up to a few KB). When you call kmalloc(100, GFP_KERNEL), SLUB gives you a slot from the 128-byte cache. This makes small allocations extremely fast — no page-boundary alignment searching needed.

C — kmalloc / kzalloc / kcalloc / krealloc — Complete Reference
#include <linux/slab.h>   /* ALL slab allocator functions live here */

/* ── kmalloc: allocate size bytes, physically contiguous, fast ─────── */
void *kmalloc(size_t size, gfp_t flags);

char *buf = kmalloc(256, GFP_KERNEL);
if (!buf)                   /* ALWAYS check for NULL */
    return -ENOMEM;
/* WARNING: Contents are UNINITIALIZED — could contain old data */
kfree(buf);                 /* Release when done */

/* ── kzalloc: kmalloc + zero initialisation (PREFER THIS) ─────────── */
void *kzalloc(size_t size, gfp_t flags);

struct my_device *dev = kzalloc(sizeof(*dev), GFP_KERNEL);
if (!dev)
    return -ENOMEM;
/* dev is zeroed: dev->id=0, dev->ptr=NULL, dev->flags=0 etc. */
/* Use kzalloc for structs — prevents info leaks, catches missing init */
kfree(dev);

/* ── kcalloc: allocate array of n elements, each size bytes, zeroed ── */
void *kcalloc(size_t n, size_t size, gfp_t flags);

int *array = kcalloc(100, sizeof(int), GFP_KERNEL);
if (!array)
    return -ENOMEM;
/* kcalloc checks for overflow in n*size — safer than kmalloc(n*size) */
kfree(array);

/* ── krealloc: resize an existing allocation ──────────────────────── */
void *krealloc(const void *p, size_t new_size, gfp_t flags);

buf = krealloc(buf, 512, GFP_KERNEL);
if (!buf)  /* original buf was freed if krealloc fails! save pointer! */
    return -ENOMEM;

/* ── kfree_sensitive: zero before freeing (for secrets/keys) ──────── */
void kfree_sensitive(const void *objp);
kfree_sensitive(crypto_key);  /* zeros memory before returning to slab */

/* ── SIZE LIMITS for kmalloc ─────────────────────────────────────── */
/* kmalloc works best up to: KMALLOC_MAX_SIZE (usually 4MB on 64-bit)
 * Practical sweet spot: < 4 KB (one page). For larger: use vmalloc.
 * kmalloc is physically CONTIGUOUS → scarce resource for large sizes
 * Check: cat /proc/slabinfo to see slab cache usage */

⚙️ How SLUB Works Internally

SLUB maintains a set of size-class caches — one cache for each power-of-2 size (and some intermediate sizes): kmalloc-8, kmalloc-16, kmalloc-32, ..., kmalloc-4096. When you call kmalloc(100, ...), it rounds up to 128 and grabs from kmalloc-128.

Each cache maintains a per-CPU cache of free objects (the "freelist"). Allocations from the per-CPU cache are lock-free and nearly instant. Only when the per-CPU cache runs dry does SLUB go to the node-level list, which needs a lock.

vmalloc / vfree — Large Virtually Contiguous Allocations

🔀 When kmalloc Isn't Enough

vmalloc() allocates memory that is contiguous in virtual address space but may be scattered in physical memory. This is the key difference — it can satisfy large allocations that kmalloc would fail on because it doesn't need physically contiguous pages.

The trade-off: vmalloc is slower (needs to set up individual page table entries), vmalloc regions are not in the direct map, and vmalloc memory cannot be used for DMA.

KMALLOC — physically AND virtually contiguous: Virtual: [ page 0 ][ page 1 ][ page 2 ] contiguous Physical: [ page 0 ][ page 1 ][ page 2 ] contiguous ↑ These physical pages are sequential → OK for DMA VMALLOC — virtually contiguous, physically scattered: Virtual: [ page 0 ][ page 1 ][ page 2 ] contiguous ← addresses sequential Physical: [ page 41 ] [ page 892 ] [ page 12 ] scattered ← DMA cannot use this! vmalloc must set up 3 separate PTEs (page table entries) to map the scattered physical pages into a contiguous virtual address range. Each page boundary access = potential TLB miss → slower than kmalloc
C — vmalloc API Complete Reference
#include <linux/vmalloc.h>

/* ── vmalloc: virtually contiguous, may sleep ──────────────────────── */
void *vmalloc(unsigned long size);

char *large_buf = vmalloc(8 * 1024 * 1024);  /* 8 MB — easy for vmalloc */
if (!large_buf)
    return -ENOMEM;
/* Contents are UNINITIALIZED */
vfree(large_buf);    /* use vfree, NOT kfree! Wrong function = oops */

/* ── vzalloc: vmalloc + zero initialised ──────────────────────────── */
void *vzalloc(unsigned long size);

char *zbuf = vzalloc(4 * 1024 * 1024);  /* 4 MB, zeroed */
if (!zbuf)
    return -ENOMEM;
vfree(zbuf);

/* ── vmalloc_to_page: convert vmalloc address → struct page * ──────── */
struct page *pg = vmalloc_to_page(large_buf + 4096);
/* Needed when you want to use vmalloc'd pages for user-space mapping
 * via vm_insert_page() — since the physical pages aren't contiguous */

/* ── is_vmalloc_addr: check if an address is from vmalloc ─────────── */
if (is_vmalloc_addr(ptr)) {
    vfree(ptr);      /* vmalloc memory */
} else {
    kfree(ptr);      /* kmalloc memory */
}

/* ── CRITICAL RULES ─────────────────────────────────────────────────
 * 1. NEVER pass vmalloc memory to DMA (physically discontiguous)
 * 2. NEVER call vfree() from interrupt context (may sleep)
 * 3. NEVER use kfree() on vmalloc memory → crash
 * 4. vmalloc() implies GFP_KERNEL — can only call from process context
 * 5. vmalloc is slower than kmalloc — don't use for small allocations */

Page Allocator — Direct Page-Level Allocation

📄 Working Directly with Pages

Below kmalloc is the page allocator (buddy allocator). It allocates in powers of 2 pages. kmalloc itself calls the page allocator for large requests. As a driver developer, you rarely call the page allocator directly — but you need to understand it because DMA operations, frame buffers, and mmap() implementations use it.

C — Page Allocator API
#include <linux/gfp.h>     /* GFP flags, alloc_page */
#include <linux/highmem.h> /* kmap, kunmap for HIGHMEM */

/* ── Allocate a SINGLE page ──────────────────────────────────────── */
struct page *pg = alloc_page(GFP_KERNEL);
if (!pg)
    return -ENOMEM;

void *vaddr = page_address(pg);   /* get kernel virtual address of the page */
memset(vaddr, 0, PAGE_SIZE);      /* PAGE_SIZE = 4096 on most architectures */
__free_page(pg);                   /* free the single page */

/* ── Allocate 2^order CONTIGUOUS pages ──────────────────────────── */
/* order=0 → 1 page (4KB), order=1 → 2 pages (8KB), order=10 → 4MB */
unsigned int order = get_order(65536);  /* calc order for 64KB = order 4 */
struct page *pages = alloc_pages(GFP_KERNEL, order);
if (!pages)
    return -ENOMEM;

void *buf = page_address(pages);   /* contiguous virtual range */
free_pages((unsigned long)buf, order);  /* free by address */

/* ── Alternative: get address directly ──────────────────────────── */
unsigned long addr = __get_free_pages(GFP_KERNEL, order);
/* addr is directly usable as a pointer — equivalent to page_address() */
free_pages(addr, order);           /* free using the address form */

/* ── Get zeroed pages ────────────────────────────────────────────── */
unsigned long zaddr = get_zeroed_page(GFP_KERNEL);
/* single zeroed page — used for user-mappable pages that must be clean */
free_page(zaddr);                  /* free single zeroed page */

/* ── IMPORTANT: PAGE_SIZE vs PAGE_SHIFT ─────────────────────────── */
/* PAGE_SIZE  = 4096 (0x1000) on x86/ARM — bytes per page
 * PAGE_SHIFT = 12          — bits to shift: 1 << PAGE_SHIFT == PAGE_SIZE
 * PAGE_MASK  = ~(PAGE_SIZE-1) — mask for page-aligning an address
 *
 * Page-align a size:  ALIGN(size, PAGE_SIZE)
 * Page-align an addr: addr & PAGE_MASK
 * Address → page number: addr >> PAGE_SHIFT  */

Memory Pools — Guaranteed Allocations in Emergency Paths

🏊 What is a Memory Pool?

A memory pool (mempool_t) pre-allocates a set of objects and holds them in reserve. When the system is under memory pressure and normal allocation would fail, the memory pool can still serve requests from its reserve. This is critical for drivers that must allocate memory in error-handling or recovery paths — if the allocation fails there, the situation gets even worse.

Block device drivers use mempools extensively — they need to allocate struct bio in I/O completion paths where sleeping isn't always safe and allocation failure is unacceptable.

C — mempool API
#include <linux/mempool.h>

static mempool_t *my_pool;

/* ── Create pool: reserve 16 pre-allocated objects ──────────────── */
static int __init driver_init(void)
{
    /* mempool_create(min_nr, alloc_fn, free_fn, pool_data)
     * The pool keeps at least min_nr elements reserved.
     * When pool is empty and normal alloc fails, uses reserves. */
    my_pool = mempool_create_kmalloc_pool(16, sizeof(struct my_request));
    if (!my_pool)
        return -ENOMEM;
    return 0;
}

/* ── Allocate from pool (can sleep if GFP_KERNEL, won't fail if reserves) */
struct my_request *req = mempool_alloc(my_pool, GFP_ATOMIC);
/* With GFP_ATOMIC: tries normal alloc first, falls back to reserves.
 * mempool_alloc with GFP_KERNEL can sleep but WILL NOT return NULL
 * as long as reserves are available. */

/* ── Free back to pool ───────────────────────────────────────────── */
mempool_free(req, my_pool);
/* Returns object to reserves first if pool is depleted, else to slab */

/* ── Destroy pool on cleanup ─────────────────────────────────────── */
static void __exit driver_exit(void)
{
    mempool_destroy(my_pool);
}

The Kernel Stack — Why It's Tiny and What to Do About It

📚 8KB Stack — Non-Negotiable Limit

Every kernel thread and every syscall has a stack of exactly 8KB (16KB with CONFIG_VMAP_STACK and KASAN). This is fixed — it cannot grow. Exceed it and you get a silent stack overflow that corrupts adjacent kernel memory — one of the nastiest kernel bugs to debug.

The kernel can detect stack overflows with CONFIG_STACK_TRACER and CONFIG_VMAP_STACK — but only if you've enabled them. In production, a stack overflow typically causes corruption that manifests as a crash far from the actual overflow.

C — Stack Usage Rules and Patterns
/* ── WRONG: Large arrays on stack — WILL OVERFLOW ──────────────── */
static int bad_function(void)
{
    char buffer[4096];        /* BAD! 4KB on stack = half your stack gone */
    int  array[1024];         /* BAD! 4KB more = stack overflow */
    struct big_struct bs;    /* BAD if sizeof(big_struct) > ~256 bytes */
    return 0;
}

/* ── CORRECT: Allocate large buffers on heap ─────────────────────── */
static int good_function(void)
{
    char *buffer = kmalloc(4096, GFP_KERNEL);
    if (!buffer)
        return -ENOMEM;

    /* use buffer... */

    kfree(buffer);
    return 0;
}

/* ── RULE OF THUMB ─────────────────────────────────────────────────
 * Keep individual stack frames under 200 bytes.
 * Stack depth in kernel can be 10-20 frames deep.
 * 20 frames × 200 bytes = 4KB consumed from nested calls alone.
 *
 * CHECK your stack usage with:
 *   make C=1 CF=-fstack-usage   (during build)
 *   scripts/stackdelta           (compare two builds)
 *
 * The kernel build warns about functions using >1024 bytes of stack:
 *   arch/x86/kernel/foo.c:42: warning: 'bar' uses too much stack */

/* ── CHECKING CURRENT STACK USAGE ─────────────────────────────────── */
void check_stack(void)
{
    /* dump_stack() prints current stack trace to dmesg — for debugging */
    dump_stack();

    /* How much stack is left? (rough estimate) */
    unsigned long sp;
    asm("mov %%rsp, %0" : "=r"(sp));
    pr_info("Stack remaining: ~%lu bytes\n",
            sp - (unsigned long)task_stack_page(current));
}

devm_ APIs — Automatic Resource Management

🧹 Device-Managed Resources — The Modern Way

The devm_ (device-managed) family of functions automatically frees resources when the device is removed or probe fails. This eliminates entire categories of resource-leak bugs. You should use devm_ functions in every platform/I2C/SPI driver you write — they are the modern, preferred API.

C — devm_ Memory Allocation vs Manual Allocation
#include <linux/device.h>   /* devm_kmalloc, devm_kzalloc */

/* ── OLD WAY: Manual allocation + goto cleanup ───────────────────── */
static int old_probe(struct platform_device *pdev)
{
    struct my_dev *dev;
    int irq, ret;

    dev = kzalloc(sizeof(*dev), GFP_KERNEL);
    if (!dev) { ret = -ENOMEM; goto err_alloc; }

    irq = platform_get_irq(pdev, 0);
    ret = request_irq(irq, my_handler, 0, "mydev", dev);
    if (ret) goto err_irq;

    return 0;

err_irq:    kfree(dev);
err_alloc:  return ret;
}

static int old_remove(struct platform_device *pdev)
{
    struct my_dev *dev = platform_get_drvdata(pdev);
    free_irq(irq, dev);
    kfree(dev);
    return 0;
}

/* ── NEW WAY: devm_ automatically frees everything ───────────────── */
static int new_probe(struct platform_device *pdev)
{
    struct my_dev *dev;
    int irq, ret;

    dev = devm_kzalloc(&pdev->dev, sizeof(*dev), GFP_KERNEL);
    if (!dev)
        return -ENOMEM;   /* no goto needed — nothing to clean up yet */

    irq = platform_get_irq(pdev, 0);
    ret = devm_request_irq(&pdev->dev, irq, my_handler, 0, "mydev", dev);
    if (ret)
        return ret;        /* dev is auto-freed by devm system */

    return 0;
}
/* remove() not needed for devm_ resources — they're freed automatically
 * when the device is unbound. No manual cleanup required. */

/* ── OTHER devm_ FUNCTIONS YOU'LL USE DAILY ─────────────────────── */
/* devm_kzalloc(dev, size, GFP_KERNEL)    — like kzalloc, auto-freed */
/* devm_kmalloc(dev, size, GFP_KERNEL)    — like kmalloc, auto-freed */
/* devm_kstrdup(dev, str, GFP_KERNEL)     — duplicate a string */
/* devm_request_irq(dev, irq, ...)        — auto free_irq on remove */
/* devm_ioremap(dev, phys, size)          — auto iounmap on remove */
/* devm_gpiod_get(dev, name, flags)       — auto gpiod_put on remove */
/* devm_clk_get(dev, name)               — auto clk_put on remove */
/* devm_regulator_get(dev, supply)        — auto regulator_put */
💡
When NOT to Use devm_Don't use devm_ if the resource lifetime doesn't match device lifetime. For example, if you allocate a buffer in your .read() file operation and free it after the read completes, that's a per-operation allocation — use plain kmalloc()/kfree(). devm_ is specifically for resources that should live as long as the device is bound to the driver.

Day 4 Practice Project — Memory Exploration Module

🎯
Today's GoalBuild a module that exercises every allocator — kmalloc, kzalloc, kcalloc, vmalloc, and pages — checks returned memory, compares performance, and deliberately tests error paths. Then use /proc/slabinfo and kmemleak to verify no leaks.
C — memory_practice.c — Full Practice Module
/*
 * memory_practice.c — Day 4: Kernel Memory Allocators
 * Tests kmalloc, kzalloc, vmalloc, page allocator
 * Run: sudo insmod memory_practice.ko
 *      sudo insmod memory_practice.ko size=8192 verbose=1
 */
#include <linux/init.h>
#include <linux/module.h>
#include <linux/kernel.h>
#include <linux/slab.h>      /* kmalloc, kzalloc, kfree */
#include <linux/vmalloc.h>   /* vmalloc, vfree */
#include <linux/gfp.h>       /* GFP flags */
#include <linux/mm.h>        /* PAGE_SIZE, alloc_pages */
#include <linux/highmem.h>   /* page_address */

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Your Name");
MODULE_DESCRIPTION("Day 4 Memory Management Practice");

static int  size = 256;
static bool verbose = false;
module_param(size, int, 0444);
module_param(verbose, bool, 0444);
MODULE_PARM_DESC(size, "Allocation size for kmalloc tests (default: 256)");
MODULE_PARM_DESC(verbose, "Print detailed allocation info");

/* Helper: print allocation details if verbose */
static void show_alloc(const char *name, void *ptr, size_t sz)
{
    if (!verbose)
        return;
    if (ptr)
        pr_info("  %-20s ptr=%px  phys=%llx  size=%zu  vmalloc=%d\n",
                name, ptr,
                (unsigned long long)virt_to_phys(ptr),
                sz, is_vmalloc_addr(ptr));
    else
        pr_err("  %-20s FAILED!\n", name);
}

static int test_kmalloc(void)
{
    void *p1, *p2, *p3;

    pr_info("--- kmalloc / kzalloc / kcalloc tests ---\n");

    /* Test 1: basic kmalloc */
    p1 = kmalloc(size, GFP_KERNEL);
    show_alloc("kmalloc", p1, size);
    if (!p1) return -ENOMEM;

    /* Test 2: kzalloc — must be zero */
    p2 = kzalloc(size, GFP_KERNEL);
    show_alloc("kzalloc", p2, size);
    if (!p2) { kfree(p1); return -ENOMEM; }

    /* Verify kzalloc really zeroed it */
    if (memchr_inv(p2, 0, size) != NULL)
        pr_err("kzalloc returned non-zero memory!\n");
    else
        pr_info("  kzalloc: memory is properly zeroed ✓\n");

    /* Test 3: kcalloc — array of 32 ints */
    p3 = kcalloc(32, sizeof(int), GFP_KERNEL);
    show_alloc("kcalloc(32,int)", p3, 32 * sizeof(int));
    if (!p3) { kfree(p1); kfree(p2); return -ENOMEM; }

    /* Show slab cache used (ksize returns actual allocation size) */
    pr_info("  ksize(p1)=%zu  ksize(p2)=%zu (rounded up by slab)\n",
            ksize(p1), ksize(p2));

    kfree(p1); kfree(p2); kfree(p3);
    pr_info("  All kmalloc memory freed ✓\n");
    return 0;
}

static int test_vmalloc(void)
{
    void *vp;
    size_t big = 2 * 1024 * 1024;  /* 2MB */

    pr_info("--- vmalloc tests ---\n");

    vp = vmalloc(big);
    if (!vp) { pr_err("vmalloc(2MB) failed\n"); return -ENOMEM; }

    pr_info("  vmalloc(2MB)=%px  is_vmalloc=%d\n",
            vp, is_vmalloc_addr(vp));

    /* vmalloc is NOT in the direct map — virt_to_phys does NOT work */
    pr_info("  virt_to_phys on vmalloc is wrong: %llx (don't trust this)\n",
            (unsigned long long)virt_to_phys(vp));

    /* Write to it to verify it's actually accessible */
    memset(vp, 0xAB, big);
    pr_info("  vmalloc memory is accessible (wrote 2MB) ✓\n");

    vfree(vp);
    pr_info("  vmalloc memory freed ✓\n");
    return 0;
}

static int test_pages(void)
{
    struct page *pg;
    void *vaddr;
    unsigned long phys;

    pr_info("--- page allocator tests ---\n");

    pg = alloc_page(GFP_KERNEL);
    if (!pg) { pr_err("alloc_page failed\n"); return -ENOMEM; }

    vaddr = page_address(pg);
    phys = page_to_phys(pg);      /* struct page → physical address */

    pr_info("  page=%px  vaddr=%px  phys=%lx  size=%lu\n",
            pg, vaddr, phys, PAGE_SIZE);

    /* Verify: virt_to_phys(page_address(pg)) == page_to_phys(pg) */
    if (virt_to_phys(vaddr) == phys)
        pr_info("  virt_to_phys == page_to_phys ✓ (direct map verified)\n");

    __free_page(pg);
    pr_info("  Page freed ✓\n");
    return 0;
}

static int __init memory_practice_init(void)
{
    int ret;

    pr_info("=== Day 4: Memory Management Practice ===\n");
    pr_info("PAGE_SIZE=%lu  sizeof(void*)=%zu\n",
            PAGE_SIZE, sizeof(void *));

    ret = test_kmalloc();
    if (ret) return ret;

    ret = test_vmalloc();
    if (ret) return ret;

    ret = test_pages();
    if (ret) return ret;

    pr_info("=== All memory tests passed! ===\n");
    return 0;
}

static void __exit memory_practice_exit(void)
{
    pr_info("memory_practice: unloaded\n");
}

module_init(memory_practice_init);
module_exit(memory_practice_exit);

Quick Reference — Choosing the Right Allocator

DECISION TREE: Which Allocator to Use? Need to allocate memory in a driver? │ ├── Are you in INTERRUPT CONTEXT or holding a SPINLOCK? │ │ │ ├── YES → Use GFP_ATOMIC flag with kmalloc │ │ kmalloc(size, GFP_ATOMIC) │ │ ⚠ May fail! Always check NULL return. │ │ ⚠ Max size: stay small (< 64KB) │ │ │ └── NO (process context) → Continue below ↓ │ ├── Is size SMALL (< ~128KB) and you need it for DMA? │ │ │ ├── YES → kmalloc / kzalloc + dma_map_single() │ │ kzalloc(size, GFP_KERNEL) [for structs/buffers] │ │ OR │ │ dma_alloc_coherent() [for DMA buffers, Day 19] │ │ │ └── NO → Continue below ↓ │ ├── Is size LARGE (> 128KB) and no DMA needed? │ │ │ └── YES → vmalloc / vzalloc │ vzalloc(size) [virtually contiguous] │ ├── Need EXACTLY N pages (e.g., for mmap to userspace)? │ │ │ └── YES → Page allocator │ alloc_pages(GFP_KERNEL, order) [2^order pages] │ ├── Is this a DEVICE DRIVER with struct device *dev? │ │ │ └── YES → Prefer devm_ variants │ devm_kzalloc(&dev, size, GFP_KERNEL) [auto-freed on unbind] │ └── Need GUARANTEED allocation (can't fail, error recovery path)? │ └── YES → Memory pool mempool_alloc(pool, GFP_ATOMIC) [uses reserves]

What is ioctl?

ioctl (Input/Output Control) is a system call that provides a flexible, device-specific control channel. When read() and write() aren't enough — when you need to configure hardware, query status, or send commands that don't map to data I/O — ioctl is the answer.

Think of it as a Swiss Army knife: one system call with an unlimited number of commands you define, each carrying its own payload type.

The Three Arguments

Every ioctl call passes three arguments from userspace to your driver:

C userspace call 
/* Userspace: ioctl(fd, cmd, arg) */
int ioctl(int fd,         /* file descriptor from open() */
          unsigned long cmd,   /* encoded command — direction + type + nr + size */
          ...);               /* arg: optional unsigned long OR pointer to struct */

/* Examples: */
ioctl(fd, RESET_BUFFER, 0);                  /* no data */
ioctl(fd, GET_TEMP, &temp);                   /* read data OUT of driver */
ioctl(fd, SET_THRESHOLD, &threshold);         /* write data INTO driver */
💡
Why not just use read/write?

read/write are for data streams. ioctl is for out-of-band control: resetting hardware, setting baud rates, querying DMA stats, flushing FIFO, setting interrupt threshold. Conceptually different operations belong in ioctl, not embedded in data streams.

02

The Four Command Macros

Linux encodes direction, type, sequence number, and data size into a single 32-bit unsigned int command value. Never use raw integers — always use these macros from <linux/ioctl.h>.

_IO(type, nr)
No data transfer
For commands that carry no argument — pure control signals like reset, flush, start, stop. The arg in the ioctl call is ignored.
NO DATA
_IOR(type, nr, datatype)
Read: kernel → userspace
Driver sends data to the caller. "R" is from the userspace perspective — the user reads from the driver. You use copy_to_user() inside.
DRIVER → USER
_IOW(type, nr, datatype)
Write: userspace → kernel
User sends data to the driver. "W" from the user's perspective — the user writes into the driver. You use copy_from_user() inside.
USER → DRIVER
_IOWR(type, nr, datatype)
Read + Write: bidirectional
Data flows both ways in a single call — user provides input, driver returns modified output in the same buffer. Common for query-and-configure operations.
BIDIRECTIONAL

Anatomy of a Command Number

The 32-bit value is laid out precisely. Knowing this layout helps you use the decoder macros _IOC_DIR, _IOC_TYPE, _IOC_NR, _IOC_SIZE for validation inside your driver.

C ioctl cmd bit layout 
/*  Bit layout of a 32-bit ioctl command (architecture-dependent but typically):
 *
 *   31       30 29   16 15      8 7        0
 *  ┌──────────┬─────────┬─────────┬──────────┐
 *  │ direction│  size   │  type   │  number  │
 *  │  2 bits  │ 14 bits │  8 bits │  8 bits  │
 *  └──────────┴─────────┴─────────┴──────────┘
 *
 *  direction: 00 = none, 01 = write (user→kernel), 10 = read (kernel→user)
 *  size:      sizeof(datatype) — kernel validates this automatically
 *  type:      magic number unique to your driver ('k', 'S', 0xA5, etc.)
 *  number:    sequential command index 0,1,2,3...
 */

#define MY_DRIVER_MAGIC  'k'   /* pick a unique magic byte for your driver */

/* Define your commands: */
#define IOCTL_RESET_BUF    _IO  (MY_DRIVER_MAGIC, 0)
#define IOCTL_GET_BUFSIZE  _IOR (MY_DRIVER_MAGIC, 1, __u32)
#define IOCTL_SET_THRESHOLD _IOW(MY_DRIVER_MAGIC, 2, __u32)
#define IOCTL_GET_STATS    _IOWR(MY_DRIVER_MAGIC, 3, struct driver_stats)

/* Always document the valid range for safety checks: */
#define MY_DRIVER_IOC_MAXNR  3
⚠️
Magic Number Uniqueness

The "type" byte (magic number) must be unique across all drivers in the system. Check Documentation/userspace-api/ioctl/ioctl-number.rst for a registry of allocated values. Using a duplicate magic means ioctls could silently match the wrong driver.

03

Implementing unlocked_ioctl

The old .ioctl callback acquired the Big Kernel Lock (BKL) — a global mutex that serialized all ioctl calls system-wide, a massive bottleneck. Since kernel 2.6.36, only .unlocked_ioctl exists. You manage your own locking.

Shared Header (ABI Contract)

Put all ioctl definitions in a header shared between kernel and userspace. This is your ABI — change it carefully.

C include/mydriver.h ← shared by kernel + userspace 
#ifndef _MYDRIVER_H
#define _MYDRIVER_H

#include <linux/ioctl.h>   /* kernel; use <sys/ioctl.h> in userspace */
#include <linux/types.h>   /* __u32, __s32 — portable fixed-width types */

struct driver_stats {
    __u32 bytes_read;
    __u32 bytes_written;
    __u32 irq_count;
    __u32 current_temp;     /* millidegrees Celsius */
};

#define MY_DRV_MAGIC         'k'

#define IOCTL_RESET_BUF       _IO  (MY_DRV_MAGIC, 0)
#define IOCTL_GET_BUFSIZE     _IOR (MY_DRV_MAGIC, 1, __u32)
#define IOCTL_SET_THRESHOLD  _IOW (MY_DRV_MAGIC, 2, __u32)
#define IOCTL_GET_STATS      _IOWR(MY_DRV_MAGIC, 3, struct driver_stats)

#define MY_DRV_IOC_MAXNR     3

#endif

Driver Private State

C mydriver.c — device state struct 
#include <linux/module.h>
#include <linux/cdev.h>
#include <linux/uaccess.h>   /* copy_to/from_user */
#include <linux/mutex.h>
#include <linux/kfifo.h>
#include "mydriver.h"

#define BUF_SIZE   1024
#define DEVICE_NAME "mydrv"

struct mydrv_dev {
    struct cdev     cdev;
    struct mutex    lock;           /* protects all fields below */
    DECLARE_KFIFO_PTR(fifo, u8);
    u32             threshold;
    struct driver_stats stats;
};

The unlocked_ioctl Handler — Full Pattern

C mydriver.c — unlocked_ioctl implementation 
static long mydrv_ioctl(struct file *filp,
                          unsigned int  cmd,
                          unsigned long arg)
{
    struct mydrv_dev *dev = filp->private_data;
    struct driver_stats stats;
    __u32 val;
    int   ret = 0;

    /* ── Step 1: Validate magic byte and command number ── */
    if (_IOC_TYPE(cmd) != MY_DRV_MAGIC)   return -ENOTTY;
    if (_IOC_NR(cmd)   > MY_DRV_IOC_MAXNR) return -ENOTTY;

    /* ── Step 2: Validate user pointer access rights ── */
    if (_IOC_DIR(cmd) & _IOC_READ)          /* driver reads = user writes TO us */
        ret = !access_ok((void __user *)arg, _IOC_SIZE(cmd));
    else if (_IOC_DIR(cmd) & _IOC_WRITE)   /* driver writes = user reads FROM us */
        ret = !access_ok((void __user *)arg, _IOC_SIZE(cmd));
    if (ret) return -EFAULT;

    /* ── Step 3: Dispatch command ── */
    switch (cmd) {

    case IOCTL_RESET_BUF:
        mutex_lock(&dev->lock);
        kfifo_reset(&dev->fifo);
        memset(&dev->stats, 0, sizeof(dev->stats));
        mutex_unlock(&dev->lock);
        pr_info("mydrv: buffer reset\n");
        break;

    case IOCTL_GET_BUFSIZE:
        /* _IOR: we send __u32 TO userspace */
        val = BUF_SIZE;
        if (copy_to_user((__u32 __user *)arg, &val, sizeof(val)))
            return -EFAULT;
        break;

    case IOCTL_SET_THRESHOLD:
        /* _IOW: we receive __u32 FROM userspace */
        if (copy_from_user(&val, (__u32 __user *)arg, sizeof(val)))
            return -EFAULT;
        if (val == 0 || val > BUF_SIZE)
            return -EINVAL;   /* validate BEFORE using! */
        mutex_lock(&dev->lock);
        dev->threshold = val;
        mutex_unlock(&dev->lock);
        break;

    case IOCTL_GET_STATS:
        /* _IOWR: read current stats out to user */
        mutex_lock(&dev->lock);
        stats = dev->stats;
        mutex_unlock(&dev->lock);
        if (copy_to_user((struct driver_stats __user *)arg,
                         &stats, sizeof(stats)))
            return -EFAULT;
        break;

    default:
        return -ENOTTY;   /* unknown command for this device */
    }

    return ret;
}
🚨
NEVER dereference userspace pointers directly

arg is a userspace address. Doing *(u32*)arg in kernel context is undefined behavior and a critical security hole. The pointer might be invalid, swapped out, or point to a mmap'd device. Always use copy_to_user() / copy_from_user() or get_user() / put_user() for scalar values.

Register in file_operations

C mydriver.c — file_operations registration 
static const struct file_operations mydrv_fops = {
    .owner          = THIS_MODULE,
    .open           = mydrv_open,
    .release        = mydrv_release,
    .read           = mydrv_read,
    .write          = mydrv_write,
    .unlocked_ioctl = mydrv_ioctl,   /* ← our handler */

};

Userspace Side — How to Call

C test_mydrv.c — userspace test program 
#include <stdio.h>
#include <fcntl.h>
#include <sys/ioctl.h>
#include "mydriver.h"   /* the SAME header — shared ABI */

int main(void) {
    int fd = open("/dev/mydrv", O_RDWR);
    if (fd < 0) { perror("open"); return 1; }

    /* 1. Reset the buffer */
    if (ioctl(fd, IOCTL_RESET_BUF, 0) < 0)
        perror("reset failed");

    /* 2. Read buffer size */
    uint32_t size;
    ioctl(fd, IOCTL_GET_BUFSIZE, &size);
    printf("Buffer size: %u bytes\n", size);

    /* 3. Set threshold */
    uint32_t thresh = 512;
    ioctl(fd, IOCTL_SET_THRESHOLD, &thresh);

    /* 4. Get stats */
    struct driver_stats stats;
    ioctl(fd, IOCTL_GET_STATS, &stats);
    printf("Reads: %u  Writes: %u  IRQs: %u  Temp: %u m°C\n",
           stats.bytes_read, stats.bytes_written,
           stats.irq_count,  stats.current_temp);

    close(fd);
    return 0;
}
04

get_user / put_user — Scalar Shortcuts

For single scalar values (int, u32, long), the kernel provides optimized helpers that avoid the overhead of a full copy_to/from_user call.

C scalar access pattern 
case IOCTL_GET_BUFSIZE: {
    /* Option A: copy_to_user (for structs or larger types) */
    __u32 val = BUF_SIZE;
    if (copy_to_user((__u32 __user *)arg, &val, sizeof(val)))
        return -EFAULT;

    /* Option B: put_user — simpler for scalar types */
    if (put_user((__u32)BUF_SIZE, (__u32 __user *)arg))
        return -EFAULT;
    break;
}

case IOCTL_SET_THRESHOLD: {
    /* Option A: copy_from_user */
    __u32 val;
    if (copy_from_user(&val, (__u32 __user *)arg, sizeof(val)))
        return -EFAULT;

    /* Option B: get_user */
    __u32 val2;
    if (get_user(val2, (__u32 __user *)arg))
        return -EFAULT;
    break;
}
📖
Rule of thumb

Use get_user()/put_user() for primitive scalars (int, u32, u64). Use copy_to/from_user() for structs and arrays. Both check the userspace address is valid and return non-zero on fault.

05

compat_ioctl — 32-bit on 64-bit Kernel

When a 32-bit userspace process runs on a 64-bit kernel, pointer sizes differ — a long is 4 bytes in userspace but 8 bytes in kernel. Struct layouts can mismatch. The kernel routes such calls to .compat_ioctl instead of .unlocked_ioctl.

C compat_ioctl implementation 
#ifdef CONFIG_COMPAT
#include <linux/compat.h>

/* If the 32-bit struct differs from the 64-bit one, define it: */
struct driver_stats32 {
    compat_u32 bytes_read;
    compat_u32 bytes_written;
    compat_u32 irq_count;
    compat_u32 current_temp;
    /* NOTE: no pointer fields here — if you had pointers they would differ! */
};

static long mydrv_compat_ioctl(struct file *filp,
                                  unsigned int  cmd,
                                  unsigned long arg)
{
    /* Case 1: Commands with no pointer args — just forward directly */
    switch (cmd) {
    case IOCTL_RESET_BUF:
    case IOCTL_GET_BUFSIZE:
    case IOCTL_SET_THRESHOLD:
        /* These only pass __u32 — same in both 32 and 64 bit */
        return mydrv_ioctl(filp, cmd, arg);

    case IOCTL_GET_STATS:
        /* Struct is same layout — safe to call directly in this case */
        return mydrv_ioctl(filp, cmd, arg);

    default:
        return -ENOTTY;
    }
}
#endif
⚠️
When structs contain pointers

If your ioctl struct has a void * or long field, it will be 4 bytes in 32-bit but 8 bytes in 64-bit. You must define a 32-bit version of the struct using compat_uptr_t and compat_long_t, then translate between them in compat_ioctl.

06

ioctl Error Codes — What to Return

Return the correct POSIX errno so userspace programs can handle failures meaningfully. This is part of your driver's ABI.

Return Value Errno When to Use
-ENOTTY 25 Wrong device — magic byte mismatch OR unknown command number. This is the "official" ioctl error for unknown commands.
-EINVAL 22 Invalid argument — the command is known but the value passed is out of range or logically invalid (e.g. threshold = 0).
-EFAULT 14 Bad address — copy_to/from_user failed because the userspace pointer is invalid or not accessible.
-EACCES 13 Permission denied — caller lacks required capability (e.g. CAP_SYS_ADMIN) for a privileged command.
-ENOMEM 12 Out of memory — ioctl tried to allocate resources and failed. Rarely needed but legitimate.
-EBUSY 16 Device busy — operation cannot proceed now (e.g. DMA transfer in progress, device locked).
-EINTR 4 Interrupted — command was waiting (e.g. mutex_lock_interruptible) and a signal arrived.
0 — Success. For _IOR commands the data was written to userspace. For _IOW the data was accepted.

Complete Working Example — Circular Buffer Driver

Building on Days 8–9, here is a complete character driver with full ioctl support. This is the kind of driver you would submit as a portfolio piece or be asked to write in an interview.

C circbuf_driver.c — complete driver with ioctl 
// SPDX-License-Identifier: GPL-2.0
// Circular Buffer Character Driver with ioctl
// Day 10 — Linux Device Driver Roadmap

#include <linux/module.h>
#include <linux/kernel.h>
#include <linux/fs.h>
#include <linux/cdev.h>
#include <linux/device.h>
#include <linux/mutex.h>
#include <linux/kfifo.h>
#include <linux/uaccess.h>
#include <linux/slab.h>
#include "mydriver.h"

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Driver Dev");
MODULE_DESCRIPTION("Circular buffer char driver with ioctl");

#define BUF_SIZE     4096
#define DEV_NAME     "circbuf"
#define DEV_CLASS    "circbuf_class"

struct circbuf_dev {
    struct cdev         cdev;
    struct mutex        lock;
    DECLARE_KFIFO_PTR(fifo, u8);
    u32                 threshold;
    struct driver_stats stats;
};

static struct circbuf_dev  *gdev;
static dev_t               dev_num;
static struct class        *dev_class;
static struct device       *dev_node;

/* ── open ── */
static int circbuf_open(struct inode *inode, struct file *filp)
{
    struct circbuf_dev *dev = container_of(inode->i_cdev,
                                              struct circbuf_dev, cdev);
    filp->private_data = dev;
    return 0;
}

/* ── read ── */
static ssize_t circbuf_read(struct file *filp, char __user *buf,
                              size_t count, loff_t *ppos)
{
    struct circbuf_dev *dev = filp->private_data;
    unsigned int copied;
    int ret;

    mutex_lock(&dev->lock);
    ret = kfifo_to_user(&dev->fifo, buf, count, &copied);
    if (!ret) dev->stats.bytes_read += copied;
    mutex_unlock(&dev->lock);

    return ret ? ret : copied;
}

/* ── write ── */
static ssize_t circbuf_write(struct file *filp, const char __user *buf,
                               size_t count, loff_t *ppos)
{
    struct circbuf_dev *dev = filp->private_data;
    unsigned int copied;
    int ret;

    mutex_lock(&dev->lock);
    ret = kfifo_from_user(&dev->fifo, buf, count, &copied);
    if (!ret) dev->stats.bytes_written += copied;
    mutex_unlock(&dev->lock);

    return ret ? ret : copied;
}

/* ── ioctl ── */
static long circbuf_ioctl(struct file *filp,
                             unsigned int  cmd,
                             unsigned long arg)
{
    struct circbuf_dev  *dev = filp->private_data;
    struct driver_stats  stats;
    __u32                val;

    if (_IOC_TYPE(cmd) != MY_DRV_MAGIC)    return -ENOTTY;
    if (_IOC_NR(cmd)   > MY_DRV_IOC_MAXNR)  return -ENOTTY;

    switch (cmd) {

    case IOCTL_RESET_BUF:
        mutex_lock(&dev->lock);
        kfifo_reset(&dev->fifo);
        memset(&dev->stats, 0, sizeof(dev->stats));
        dev->threshold = BUF_SIZE / 2;
        mutex_unlock(&dev->lock);
        break;

    case IOCTL_GET_BUFSIZE:
        val = BUF_SIZE;
        return put_user(val, (__u32 __user *)arg) ? -EFAULT : 0;

    case IOCTL_SET_THRESHOLD:
        if (get_user(val, (__u32 __user *)arg))   return -EFAULT;
        if (val == 0 || val > BUF_SIZE)           return -EINVAL;
        mutex_lock(&dev->lock);
        dev->threshold = val;
        mutex_unlock(&dev->lock);
        break;

    case IOCTL_GET_STATS:
        mutex_lock(&dev->lock);
        stats = dev->stats;
        stats.bytes_read   = dev->stats.bytes_read;
        mutex_unlock(&dev->lock);
        if (copy_to_user((struct driver_stats __user *)arg,
                         &stats, sizeof(stats)))
            return -EFAULT;
        break;

    default:
        return -ENOTTY;
    }
    return 0;
}

static const struct file_operations circbuf_fops = {
    .owner          = THIS_MODULE,
    .open           = circbuf_open,
    .read           = circbuf_read,
    .write          = circbuf_write,
    .unlocked_ioctl = circbuf_ioctl,
};

/* ── init ── */
static int __init circbuf_init(void)
{
    int ret;

    gdev = kzalloc(sizeof(*gdev), GFP_KERNEL);
    if (!gdev) return -ENOMEM;

    ret = kfifo_alloc(&gdev->fifo, BUF_SIZE, GFP_KERNEL);
    if (ret) goto err_kfree;

    mutex_init(&gdev->lock);
    gdev->threshold = BUF_SIZE / 2;

    ret = alloc_chrdev_region(&dev_num, 0, 1, DEV_NAME);
    if (ret) goto err_fifo;

    cdev_init(&gdev->cdev, &circbuf_fops);
    gdev->cdev.owner = THIS_MODULE;
    ret = cdev_add(&gdev->cdev, dev_num, 1);
    if (ret) goto err_chrdev;

    dev_class = class_create(THIS_MODULE, DEV_CLASS);
    if (IS_ERR(dev_class)) { ret = PTR_ERR(dev_class); goto err_cdev; }

    dev_node = device_create(dev_class, NULL, dev_num, NULL, DEV_NAME);
    if (IS_ERR(dev_node)) { ret = PTR_ERR(dev_node); goto err_class; }

    pr_info("circbuf: registered /dev/%s (major %d)\n", DEV_NAME, MAJOR(dev_num));
    return 0;

err_class:  class_destroy(dev_class);
err_cdev:   cdev_del(&gdev->cdev);
err_chrdev: unregister_chrdev_region(dev_num, 1);
err_fifo:   kfifo_free(&gdev->fifo);
err_kfree:  kfree(gdev);
    return ret;
}

/* ── exit ── */
static void __exit circbuf_exit(void)
{
    device_destroy(dev_class, dev_num);
    class_destroy(dev_class);
    cdev_del(&gdev->cdev);
    unregister_chrdev_region(dev_num, 1);
    kfifo_free(&gdev->fifo);
    kfree(gdev);
    pr_info("circbuf: unregistered\n");
}

module_init(circbuf_init);
module_exit(circbuf_exit);

Production Best Practices

  • Always validate with _IOC_TYPE and _IOC_NR first. Reject unknown commands with -ENOTTY before touching any data.
  • Never use BKL-era .ioctl. Always use .unlocked_ioctl. The BKL was removed in 2.6.39.
  • Shared header = shared ABI. Put all struct and macro definitions in a header both kernel and userspace include. Never duplicate or inline.
  • Validate input before use. Range-check values from copy_from_user before passing to hardware or using as array indices.
  • Always protect shared state with a mutex. Even a simple flag needs protection — two CPUs can call ioctl simultaneously.
  • Use mutex_lock_interruptible() in ioctl handlers. Return -EINTR if a signal arrives while waiting. Allows Ctrl+C to work in userspace.
  • Implement .compat_ioctl if your driver will run on x86_64 systems where 32-bit binaries may call it.
  • Keep ioctl handlers focused. Each command should do one thing. Complex logic belongs in separate functions called from the switch.
  • Add capability checks for privileged operations. Use capable(CAP_SYS_ADMIN) for commands that configure hardware or affect system-wide state.
  • Document every command in a comment block. State direction, argument type, possible return values, and side effects.

Interview Questions — ioctl

What is the difference between .ioctl and .unlocked_ioctl?
The original .ioctl callback acquired the Big Kernel Lock (BKL) before calling your handler, serializing all ioctl calls across the entire kernel — a massive SMP scalability bottleneck. .unlocked_ioctl (introduced in 2.6.11, mandatory since 2.6.39) removes this global lock entirely. You are responsible for your own fine-grained locking using mutex, spinlock, or other primitives as appropriate for your driver.
Why use _IO/_IOR/_IOW macros instead of raw integers for command numbers?
The macros encode the direction, data size, and a magic "type" byte into the 32-bit command value. The kernel uses _IOC_SIZE(cmd) for automatic argument size validation. The magic byte prevents accidental command number collisions between unrelated drivers. It also makes code self-documenting — you can immediately see whether a command reads, writes, or transfers no data.
When would you return -ENOTTY vs -EINVAL from ioctl?
Return -ENOTTY ("not a typewriter") when the command is unknown to your driver — either the magic byte doesn't match or the number is out of range. This tells userspace "this file descriptor does not support this command." Return -EINVAL when the command is recognized but the argument value is logically invalid (e.g. a threshold of 0 or greater than buffer size).
Why can't you just dereference the arg pointer directly in kernel ioctl?
The arg is a userspace virtual address. Directly dereferencing it in kernel space is dangerous for several reasons: the page may not be present (swapped out), the address may be invalid, it may map to MMIO that would fault differently, or an attacker could race to change it between validation and use (TOCTOU). copy_from_user() / copy_to_user() handle page faults safely, verify the address is accessible, and prevent kernel privilege escalation via bad pointers.
What is compat_ioctl and when is it called?
When a 32-bit process runs on a 64-bit kernel, the kernel calls .compat_ioctl instead of .unlocked_ioctl. Pointer and long fields in structs are 4 bytes wide in 32-bit userspace but 8 bytes in the 64-bit kernel, so struct layouts may not match. In compat_ioctl you use compat_uptr_t and compat_long_t to translate between the two representations before forwarding to the main handler. If your structs only contain fixed-width types (__u32, __s64), you can often just forward directly.
Walk me through writing an ioctl from scratch — what steps in order?
1) Choose a unique magic byte. 2) Define a shared header with the struct(s) and command macros using _IO/_IOR/_IOW/_IOWR. 3) Implement the handler: validate magic/nr, check access_ok, switch on command, use copy_to/from_user, handle errors. 4) Register in file_operations as .unlocked_ioctl. 5) Implement .compat_ioctl for 32-bit compatibility. 6) Write a userspace test program using <sys/ioctl.h> and the shared header.

Quick API Reference

C ioctl API cheat sheet 
/* ── Macro constructors ── */
_IO  (type, nr)              // no data
_IOR (type, nr, datatype)   // kernel → user (user reads)
_IOW (type, nr, datatype)   // user → kernel (user writes)
_IOWR(type, nr, datatype)  // bidirectional

/* ── Decoder macros ── */
_IOC_TYPE(cmd)              // extract magic byte
_IOC_NR(cmd)                // extract sequence number
_IOC_SIZE(cmd)              // extract data size in bytes
_IOC_DIR(cmd)               // extract direction flags

/* ── Userspace pointer access ── */
copy_to_user(to, from, n)    // kernel → user, returns 0 on success
copy_from_user(to, from, n)  // user → kernel, returns 0 on success
put_user(val, ptr)           // scalar: kernel → user
get_user(val, ptr)           // scalar: user → kernel
access_ok(addr, size)        // validate userspace address (optional manual check)

/* ── 32-bit compat helpers ── */
compat_ptr(uptr)             // convert compat_uptr_t to void *
ptr_to_compat(ptr)           // convert pointer to compat_uptr_t

/* ── Capability check ── */
capable(CAP_SYS_ADMIN)       // returns true if caller has CAP_SYS_ADMIN
